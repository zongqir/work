# AES 通知配置需求文档

## 需求背景

当前 AES 已经存在多类通知场景，也涉及多种通知通道，需要解决如下问题：

- 避免通知配置入口分散
- 解决不同通知场景的配置方式不一致，理解和维护成本高的问题
- 高危消息、低危消息、汇总消息缺少统一的推送策略承接方式
- 收到无关或高频噪音消息后，缺少快速调整策略的路径
- 配置变更、消息发送结果和失败原因缺少统一追踪


## 需求识别结论

这次识别出的需求主要分两类：

- 业务需求：让不同管理员能围绕具体消息类型完成订阅、接收、处置和调整
- 非业务需求：让消息推送能力本身稳得住，也便于扩展、排查和治理

整体方向不是做一个很重的工作流平台

这里有一条边界需要先说清楚：业务语义由各业务模块负责定义，消息中心不负责定义什么是业务事件、以及业务结果如何计算。
消息中心负责把业务方已经定义好的消息类型纳入统一配置、统一发送和统一治理。

这里还要再补一条接入边界：

- 巡检类场景更适合由巡检或监控侧先产出标准化结果，再交给消息中心承接和发送
- 告警类场景如果业务侧已经有成熟的标准查询或聚合接口，可以由消息中心按配置调用接口拿结果，但告警语义、筛选逻辑和聚合逻辑仍由告警侧负责
- 不管哪种接入方式，消息中心都不负责定义业务规则，只负责承接结果后的统一配置、统一发送和统一治理

之所以要区分这两种接法，是因为两类业务形态不同：

- 巡检类更像按周期运行后产出一轮结果，结果本身天然是这一轮检查的产物
- 告警类更像持续产生的事件流，很多时候需要结合通知配置按窗口查询和聚合


## 系统边界

AES 通知配置系统是 AES 上层的通知能力，不单独建设一套完整的平台底座能力。和 XDR 底座之间的边界需要先说清楚。

### XDR 底座负责

- 提供 AES 依赖的通用底座能力，包括登录、用户、租户、授权、权限和底层通知通道等

### AES 通知配置系统负责

- 在底座提供的登录、租户、权限上下文之上，管理消息类型和通知配置
- 基于租户上下文读取和生效通知配置
- 维护消息发送策略、频率、接收对象、模板映射等通知域配置
- 承接业务方产出的通知结果，或按约定调用业务方标准接口获取标准化结果
- 负责统一发送、统一治理、统一观测和统一审计

### 明确不由 AES 通知配置系统负责的内容

- 不负责建设高复用、且与底座模块高度一致的通用能力
- 这类能力应由底座模块统一提供支持，AES 通知配置系统直接复用

例如租户、登录、用户、授权、底层通道等消息通知能力，虽然会在通知场景中被使用，但仍属于底座范围

## 业务需求识别

### 典型业务场景

下面这些场景主要用于说明通知配置系统需要覆盖的业务主题，具体系统能力要求统一收敛到后文“业务需求清单”。

#### 场景一：高危威胁与安全事件

用户：安全管理员
用户何时需要：终端发生高危威胁时
用户要达成什么目的：及时研判风险并闭环处置
用户需要什么方式：通过企微、Webhook、邮件收到实时或高频汇总消息，点击链接直达事件详情页

覆盖明细：
- 检出安全告警或安全事件

补充说明：

- 该场景需要同时支持实时推送和按 X 小时聚合汇总两类模式
- 实时推送需要避免低危、低可信度或高噪音信息误入；按时间窗口聚合的能力由业务模块提供，AES 只关注接口承接

#### 场景二：异步任务状态变化

用户：安全管理员
用户何时需要：查杀、升级等耗时任务执行中或完成时
用户要达成什么目的：确认动作是否生效，失败时及时干预
用户需要什么方式：收到任务执行状态和执行结果消息，点击进入任务详情页

覆盖明细：

- 病毒查杀任务
- 升级任务

#### 场景三：运维与平台异常

用户：IT 运维管理员、系统管理员、安全管理员
用户何时需要：终端出现大面积运维异常，或平台资源告急、组件异常、授权即将到期时
用户要达成什么目的：及时发现异常并进入排查处理，恢复终端防护能力，保障 AES 平台自身稳定运行和业务连续
用户需要什么方式：收到运维或平台异常推送，进入客户端管理、排障页面或平台对应页面处理

覆盖明细：

- 客户端大面积离线
- 升级失败
- 端点资源占用持续过高
- 服务器磁盘不足
- 服务器 CPU/MEM 异常

进一步动作：

- 排障和恢复防护能力
- 保障系统稳定与业务连续

补充说明：

- 巡检或监控侧负责异常识别、阈值判断和结果产出，消息通知服务负责结果后的通知配置和发送治理

#### 场景四：账号与角色通知

用户：系统管理员
用户何时需要：账号登录、账号授权即将过期或角色相关通知需要统一配置时
用户要达成什么目的：完成账号和角色类通知的统一配置、查询和下发协同
用户需要什么方式：前端统一请求 AES，AES 整合配置后转发给 XDR，最终由 XDR 底座能力承接生效

覆盖明细：

- 平台登录
- 账号授权过期
- 角色通知场景

补充说明：

- 该场景依赖 XDR 底座公共能力，前端统一请求 AES，再由 AES 整合后转发 XDR
- 具体协同方式、筛选条件口径和生效规则统一见后文“与 XDR 现有能力的配置协同”

#### 场景五：收到不想要的消息

用户：任意管理员
用户何时需要：收到与自己无关或过于频繁的消息时
用户要达成什么目的：关闭推送或调整推送策略
用户需要什么方式：从消息中直接跳转到配置页面，修改接收人、频率或关闭订阅

### 业务需求清单

前面的场景主要描述业务语义，核心系统能力统一收敛为下面这些。

#### 统一的消息类型管理视角

系统需要按消息类型组织通知配置，而不是把配置分散到不同菜单中。

- 提供统一的通知配置入口
- 以业务可理解的消息类型作为主视角
- 至少覆盖安全事件、任务与响应、运维与系统通知、账号与角色通知几类消息

#### 面向消息类型的订阅配置能力

系统需要允许用户围绕具体消息类型配置自己的推送策略。

- 支持按消息类型启用或关闭通知，并返回该类型实际支持的渠道列表
- 支持配置发送渠道、接收对象、推送频率和场景扩展条件
- 接收对象应优先从可选列表中选择；对于邮箱、短信等通道，需要支持选择对应的用户、邮箱或手机号信息

#### 与 XDR 现有能力的配置协同

有些配置项在 AES 侧展示和提交，但底层仍由 XDR 承接。

- 协同范围主要包括依赖 XDR 用户信息的接收对象配置，以及账号登录、账号授权过期、角色通知这类底座场景
- AES 负责统一入口、配置展示、参数整合、请求转发和配置副本查询；XDR 负责查询、变更、用户信息列表和实际承接能力
- AES 展示和提交的字段以 XDR 公共接口为准；只有 XDR 调用成功后才更新本地结果，失败时直接提示，页面只展示已同步成功的配置

#### 消息配置列表查询能力

系统需要提供统一的消息配置列表，方便用户查看当前有哪些消息类型、分别如何配置、当前配置发给谁。

- 提供统一列表页，至少展示消息类型、分类、启用状态、发送渠道、接收对象、最近修改时间和修改人
- 支持按消息分类、消息类型、启用状态、发送渠道筛选，并支持按用户名、角色名、邮箱、Webhook 标识等“发给谁”信息搜索
- 筛选与搜索字段需要统一定义；当前阶段可先不分页，后续随数据量增长再补分页

#### 不同场景的差异化配置能力

不同业务场景对通知条件的关注点不同，系统需要支持场景扩展配置。

- 安全事件、任务、运维、系统、账号等不同场景需要支持各自的扩展筛选条件
- 其中安全事件和运维场景至少支持资产范围，且资产范围需要同时支持资产 ID 和资产组ID
- 当业务结果只提供资产 ID 时，需要有统一的资产归属判断能力来判断其是否属于目标资产组ID，这类能力应尽量复用公共能力

#### 频率控制与聚合能力

不同消息的打扰强度不同，系统需要支持按消息特性配置实时或聚合推送。

- 高危安全事件需要支持实时推送和按 X 小时聚合汇总两类模式，且实时推送要有规则校验，避免低危、低可信度或高噪音消息误入
- 按时间窗口聚合的能力由业务模块提供，AES 只负责按接口承接和发送；聚合消息需要能展示摘要和示例
- 系统预警支持提前提醒或固定时间提醒，巡检频率与通知频率需要解耦

#### 可闭环的消息内容

消息不只是提醒，还要帮助用户理解和行动。

- 消息内容需要包含标题、时间、摘要和关键上下文，并支持跳转到业务详情页或当前消息类型的配置页
- 不同场景可携带不同上下文字段，实时消息和聚合消息也可以采用不同的跳转方式

#### 模板与语言适配

模板由后台维护，但不同通道的模板形态不完全一致，需要区分处理。

- 模板不是用户配置项，且模板映射必须建立在消息类型支持的渠道列表之上
- 企业微信、Webhook、邮件按文案模板处理，发送前按消息类型、通道和租户语言定位模板
- 短信需要特化处理：先定义短信模板生成 `templateCode`，发送时下发 `templateCode + value`；当前阶段仅支持单一语言模板，后续是否支持多语言取决于供应商和平台能力
- 模板缺失时需要有明确处理规则

#### 默认配置、租户覆盖与生效视图

系统需要同时支持默认配置和租户差异化配置，并让用户能看到当前真正生效的配置结果。

- 系统需要支持“默认配置 + 租户覆盖”两层模型，租户未单独配置时回退默认值（需要定义清楚每个通知的默认值信息是怎样的 每个可能默认关？比如XDR的 默认应该是关着）
- 用户查看时应直接看到当前最终生效值，并能区分哪些字段来自默认配置、哪些来自租户覆盖
- 配置变更后需要在合理时间内完成生效更新；依赖 XDR 生效的配置只展示同步成功后的结果

#### 配置变更立即生效

用户修改配置后，需要明确知道新配置何时生效。

- 配置保存后立即对后续未发送记录生效，已进入发送过程的记录按本次发送快照执行；不再满足条件的记录可以跳过发送
- 用户修改推送频率时，需要明确提示本次修改会影响哪些发送结果，以及是否会导致合并、延后、提前发送或取消
- 巡检类场景下，已按旧业务条件产出的结果不回改，只影响后续新的巡检结果

#### 基础权限与审计能力

通知配置属于高影响配置，需要有基础权限控制和变更追溯。

- 至少区分查看和修改两类权限（TODO 这里目前看我们组的通知没有做）
- 需要记录配置修改人、修改时间、修改前后差异、所属消息类型、所属租户，以及消息发送结果和失败原因

## 非业务需求识别

### 执行稳定性需求

通知执行链路需要具备稳定的执行治理能力，避免重复发送、失败失控、限流误判和配置漂移。

- 系统需要具备贯穿 AES 执行层和下游通道调用链路的幂等能力，能够识别重复发送，并按消息类型支持不同粒度的幂等策略
- 发送失败后需要支持有限次重试，重试次数、间隔和封顶值可配置；达到上限后进入明确失败状态，并保留可追溯的结果记录
- 系统需要识别下游限流或容量保护信号，并与普通失败分开处理，支持记录、监控和退避/延后
- 执行时必须基于当前有效配置发送，尚未发送的记录按发送时配置执行，已进入发送过程的记录按发送快照执行
- 系统需要保留成功、失败、失败原因等关键事实，并对巡检类延迟发送场景区分“发送配置变更”和“业务结果口径变更”

### 资源隔离与容量控制需求

不同类型消息的执行特性不同，系统需要考虑执行资源隔离和容量控制，避免互相影响。

- 高时效消息和后台批量消息需要避免相互阻塞
- 系统需要具备基本的并发控制和下游承载保护能力，避免集中发送打爆下游通道

### 可维护性需求

系统不能越做越散，后面新增消息类型时也不能每次都重新搭一套。

- 每种消息类型当前只保留一套全局配置，配置来源清晰，可区分默认值和当前生效值
- 新增消息类型时不应重建独立页面，基础配置和扩展配置需要分层承接
- 模板由后台统一维护，模板、通道和语言适配关系需要保持可扩展

### 可观测与可追溯性需求

消息推送能力需要具备基础监控和观测能力，便于发现下游调用失败、发送异常、积压和配置问题。

- 需要具备配置变更审计、发送结果记录和失败原因回溯能力，使用户和运维能够判断一条消息是按什么配置发出的
- 需要统计各消息类型和各通道的发送量、成功率、失败率，并监控下游调用失败、超时、异常返回以及聚合任务和待发送记录积压
- 需要暴露 Prometheus 可采集指标，并能结合日志、指标和发送结果定位问题


### 安全与权限需求

通知配置属于高影响配置，系统需要具备基础的访问控制能力。

- 至少区分查看和修改两类权限，只有具备修改权限的用户才能调整消息配置
- 配置修改行为需要纳入审计，消息中的配置跳转链接也必须在登录和鉴权后才能执行修改

### 兼容性需求

系统需要适配当前计划支持的多种通道和多语言场景。

- 至少兼容企业微信、Webhook、邮件三类通道，不同通道允许适配不同内容格式
- 系统需要兼容平台现有多语言口径；短信是否支持多语言单独按通道能力处理

## 当前需求边界

### 本次纳入

- 统一的通知配置入口
- 基于消息类型的列表、详情和编辑页
- 基础配置与场景扩展配置
- 默认配置、租户覆盖与最终生效视图
- 频率控制与聚合配置
- 消息详情跳转和配置跳转
- 模板映射与语言适配规则
- 配置生效规则
- 幂等、有限重试和限流处理
- 配置变更审计与发送结果记录
- 监控指标、日志和基础告警
- 租户粒度灰度验证能力


## 待明确项

- 企业微信、Webhook、邮件三类通道在当前阶段是否全部上线（webhook考虑）
- 短信通道的 `templateCode + value` 结构、参数校验规则和供应商对接边界
- 短信当前是否明确只支持单一语言，以及后续是否需要支持海外短信和多语言模板
- 资产范围筛选在涉及资产组时，由业务方先完成归属判断并输出结果，还是由 AES 基于资产 ID 调用统一资产能力反查判断
- 如果由 AES 侧做资产组归属判断，需要由哪一层提供统一查询能力，以及查询时点是在配置保存阶段、结果进入阶段还是发送匹配阶段
- 告警类场景是否采用“消息中心主动调用业务侧标准接口拿结果”的模式
- 巡检类场景是否统一采用“业务侧先产结果，消息中心承接后发送”的模式
- 高危威胁与安全事件在按 X 小时聚合汇总时的标准接口字段和返回格式
- 重试次数、重试间隔和人工介入方式如何设定
- 配置缓存的生效时延和是否需要手动刷新能力
- Prometheus 指标和告警阈值
