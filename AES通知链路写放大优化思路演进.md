# AES 通知链路写放大优化思路演进

## 1. 为什么要单独讨论写放大

这次方案讨论到后面，真正反复卡住我们的，其实不是：

- 要不要分层
- 要不要双通道
- 要不要二方库

而是：

- 写放大

因为只要通知链路继续走：

- DB 存调度
- DB 存 batch
- DB 存发送结果

那数据库写放大就会变成系统性的风险。

所以这部分不是一个局部优化点，而是整个方案是否能落地的核心约束。

## 2. 一开始的直觉方案

最开始如果顺着“状态要完整、调度要精确、任务要可追踪”的思路往下走，很容易得到一版看起来很完整的设计：

- 每个租户每类通知维护自己的调度状态
- 每次执行完都更新 `next_run_at`
- 配置变更后批量刷新大量租户状态
- 业务层产出中间过程记录
- 执行器推进 `pending -> sending -> success/failed`
- 所有过程都尽量落库

这版设计的好处是：

- 状态很全
- 看起来很可控
- 每一步都能追踪

但问题也很快暴露：

- 一条通知会引出多次写
- 一个租户组有很多租户时，调度状态写会被放大
- 配置变更会带来大量额外写
- 中间态写非常多

也就是说，这版“完整状态设计”虽然看起来严谨，但很容易把系统写重。

## 3. 第一轮收缩：接受 at-least-once

后面一个很重要的转折是：

- 接受重复下发
- 依赖下游 `notification` 做幂等

这个转折非常关键，因为它意味着：

- 上游不再需要为了“绝不重复”维护很多细粒度状态

这一步带来的直接变化是：

- 不需要过度追求细颗粒度执行进度
- 中间过程可以允许重跑
- 只要幂等 key 稳定，重复下发就不是致命问题

也就是说，系统从：

- exactly-once 幻觉

转成了：

- at-least-once + 下游幂等

这是写放大优化的第一个根本前提。

## 4. 第二轮收缩：调度从租户级往组级收

接下来暴露出来的更大问题是：

- 如果按租户级维护调度状态
- 一个租户组有 `1000` 个租户
- 那就会天然出现 `1000` 份时间状态、`1000` 次推进写

所以第二轮收缩的核心是：

- 不再默认按租户维护调度状态
- 尽可能按租户组维护调度状态

这一步带来的变化是：

- 不给每个租户都写一份 `next_run_at`
- 同一组共享调度推进
- 真正的租户级写，尽量推迟到“有消息才落 batch”的阶段

同时还保留一个例外：

- 对少量特殊租户 override，才考虑更细粒度处理

也就是说，大部分租户走组级，特殊租户才下沉。

这是降写的第二个大拐点。

## 5. 第三轮收缩：业务只写最终 batch，不写过多中间表

再往下，一个明显的问题是：

- 业务生产阶段如果拆太多中间表
- 会很快形成额外写放大

例如：

- 先写候选表
- 再写汇总表
- 再写待发送表

所以第三轮收缩定下来的原则是：

- 业务层直接产出最终待发送 batch

也就是说：

- 查数据
- 聚合
- 有结果才写 batch

而不是：

- 先把中间过程全部状态化

这个决定又砍掉了一层很重的中间写。

## 6. 第四轮收缩：只对“有内容的租户”写 batch

调度按组推进之后，另一个自然问题是：

- 如果一组 1000 个租户都到点了
- 难道要给 1000 个租户全写一遍 batch 吗

这里后面的收敛是：

- 调度按组触发
- 数据查询可以按组查
- 但只有真正有内容的租户才写 batch

也就是说：

- 到点不等于一定写
- 有数据、有通知内容，才真正落记录

这一步把写放大的焦点进一步往“真实有通知的租户”上收。

## 7. 第五轮收缩：batch 颗粒度做大，不拆太细

当 batch 已经成为主要写入点后，接下来的优化重点就不再是“写不写”，而是“怎么少写”。

这一轮的主要收敛是：

- 一租户一窗口尽量只写一条 batch
- 不因为渠道不同就拆很多条
- 尽量让执行器去处理渠道渲染，而不是业务层先拆成多条发送记录

这一步背后的原则是：

- batch 写是必要写
- 但必要写也要尽量粗粒度

所以 batch 层的优化重点变成：

- 少拆分
- 少重复
- 少写空记录

## 8. 第六轮收缩：执行器不强依赖 `sending`

执行器这边一开始也很容易走向：

- `pending`
- `sending`
- `success`
- `failed`

甚至更多状态。

但后面讨论清楚之后，发现如果当前模型是：

- `aes-crontab` 多实例部署
- 但分布式锁单活

那么第一版完全没必要为了“看起来状态完整”去强依赖很多中间态。

于是这一轮收缩的方向变成：

- 弱化 `sending`
- 强化最终结果
- 接受执行过程中断后重跑

这里的核心前提仍然是：

- 下游幂等

这又减少了一类中间写。

## 9. 第七轮收缩：幂等 key 设计成为核心

当我们明确接受：

- 重跑
- 重复下发
- 上游状态尽量轻

之后，幂等 key 就从“一个小细节”变成了整个方案能不能成立的核心。

后面最终收敛成：

- 定时/汇总型通知按窗口做幂等
- 响应式通知按事件做幂等

例如：

- 定时型：`messageType + tenantId + windowStart`
- 响应式：`messageType + tenantId + bizId/eventId + eventSemantic`

这一步的意义在于：

- 让系统不需要通过大量状态写来避免重复
- 而是通过稳定幂等键把“重跑”变成可接受动作

## 10. 第八轮收缩：调度状态不追求“未来时间点全量维护”

一开始很容易觉得：

- 调度系统必须精确维护所有租户的 `next_run_at`

但后面这个点也开始被收缩：

- 不要为了“未来某个时间点”维护过多状态
- 更关注最小必要推进

最终思路变成：

- 组级推进优先
- 尽量减少大量租户级未来时间点写
- 默认配置变更也不要轻易做全量刷新

这一步虽然还没有完全定死成唯一方案，但方向已经很明确：

- 调度状态不能设计成高写放大的大表模型

## 11. 到目前为止，哪些写被认定为必要写

经过前面这些收缩后，当前比较明确的必要写只剩下这些：

### 11.1 batch 写

代表：

- 这条通知真的产生了

### 11.2 最终结果写

代表：

- 这条通知最终发成还是失败

### 11.3 少量调度推进写

代表：

- 这一组这类通知本轮已经跑过

也就是说，写放大并没有完全消失，但已经被明显收缩到了几个关键事实点。

## 12. 哪些写被逐步认定为“应该避免”

到目前为止，已经比较明确不该继续扩大的写包括：

- 每租户细粒度 `next_run_at`
- 配置变更导致的大量全量刷新
- 候选表、汇总表、待发送表多层中间表
- 执行器过多中间状态
- 没有内容也写 batch
- 同一窗口按渠道过度拆分 batch

## 13. 当前版本的核心优化思路

把前面这些演进收起来，当前这版通知链路的降写逻辑可以概括成下面几条：

1. 接受 at-least-once，不追求上游绝对不重复
2. 依赖下游幂等，稳定设计幂等 key
3. 调度优先按租户组推进，而不是默认逐租户推进
4. 只有真正有内容的租户才写 batch
5. batch 尽量粗粒度，一租户一窗口尽量一条
6. 执行器尽量少写中间态，强化最终结果写
7. DB 只存关键事实，不存过多中间过程

## 14. 这套演进的意义

这套思路演进最重要的变化，不是“某张表少了几个字段”，而是整体观念变了。

从一开始更偏向：

- 状态完整
- 过程全追踪
- 未来时间点强维护

逐步转向：

- 关键事实落库
- 中间过程尽量内存化或动态计算
- 允许重跑
- 用幂等替代细粒度状态

这才是这次写放大优化真正的核心收获。

## 15. 当前结论

这次关于写放大的讨论，实际上推动整个通知方案完成了一次很大的收缩。

一句话总结：

AES 通知链路的降写思路，不是“某个地方再少写一条 SQL”，而是通过“组级调度、batch 粗粒度、下游幂等、弱化中间态”把系统从“状态驱动”逐步收缩成“关键事实驱动”。
