# 多服务场景下的 AES 执行器设计

## 1. 背景

当前 AES 不是单体服务，而是多个业务服务共同参与通知场景。

这会直接带来一个关键问题：

- 业务数据分散在不同服务里
- 但通知发送能力又不适合每个服务都各自实现一套

因此，在多服务场景下，通知链路不能简单做成“每个服务自己查数据、自己组装、自己发消息”，否则会很快出现重复建设和治理失控的问题。

## 2. 推荐结论

在多服务场景下，推荐采用：

- 分散生产
- 集中发送

也就是说：

- 各业务服务负责产出待发送批次
- 一个统一执行器负责真正发送

一句话总结：

多服务下最合理的模型，不是“每个服务都发”，而是“每个服务产批次，一个公共执行器发消息”。

## 3. 总体架构

建议整体链路按下面这条路径落地：

1. 编排层提供通知规则
2. 各业务服务按规则调度自己的业务查询
3. 各业务服务产出标准化待发送批次
4. 统一写入一张待发送批次表
5. 公共执行器扫描批次表
6. 执行器渲染模板并调用 `notification HTTP`
7. 更新发送状态

这里的关键分工是：

- 编排层定规则
- 业务服务产批次
- 执行器发消息

## 4. 为什么不建议每个服务都自己发

如果每个服务都自己做发送，会有几个直接问题：

- 每个服务都要重复实现模板渲染
- 每个服务都要重复实现发送状态和重试
- 每个服务都要自己控制并发和限流
- 下游 `notification` 的调用压力无法统一治理
- 故障排查时很难快速定位到底是哪个服务发的

最终结果通常是：

- 第一版能跑
- 第二版开始越来越散
- 后面改一处发送逻辑，要改多个服务

所以在多服务场景下，“分散发送”通常不是一个好边界。

## 5. 各角色职责

### 5.1 编排层

编排层负责：

- 维护通知默认配置
- 提供租户覆盖配置
- 返回最终生效配置
- 规定频率、渠道、额度、模板绑定

编排层不负责：

- 查询具体业务数据
- 直接发送消息

### 5.2 业务服务

每个业务服务负责：

- 读取编排层的最终生效配置
- 按规则执行自己的业务调度
- 查询本服务掌握的业务数据
- 做筛选、聚合、汇总
- 组装标准化待发送批次
- 把批次写入统一批次表

业务服务不负责：

- 模板渲染
- 调 `notification`
- 控制下游发送节奏

### 5.3 公共执行器

执行器负责：

- 扫描待发送批次表
- 抢占待发送记录
- 注入模板并渲染消息内容
- 控制固定并发
- 控制发送速率和节流
- 调用 `notification` 二方库
- 记录发送结果

执行器不负责：

- 查询原始业务数据
- 决定本轮哪些租户应产出批次

## 6. 推荐的数据流

数据流建议如下：

### 6.1 业务服务产批次

每个业务服务内部有自己的生产调度逻辑。

例如：

- 安全服务定时汇总安全告警
- 审计服务定时汇总管理员登录和任务状态变更
- 系统服务定时扫描授权阈值和剩余时间

这些服务在产出结果后，统一写入一张公共批次表，例如：

- `aes_notify_batch`

### 6.2 执行器消费批次

执行器只关心这张表里的标准化批次记录。

执行器不需要知道：

- 这条批次来自哪个业务表
- 原始数据怎么查出来的
- 汇总逻辑为什么这样

它只需要知道：

- 这是一条待发送批次
- 它应该按哪些渠道发
- 它应该用哪个模板渲染

## 7. 待发送批次表示例

表名建议：

- `aes_notify_batch`

建议字段：

- `id`
- `tenant_id`
- `biz_type`
- `message_type`
- `window_start`
- `window_end`
- `channel_set`
- `payload_snapshot`
- `template_code`
- `idempotent_key`
- `status`
- `retry_count`
- `last_error`
- `next_retry_at`
- `created_at`
- `updated_at`

字段说明：

- `biz_type`：标识来自哪个业务服务
- `message_type`：消息类型
- `payload_snapshot`：业务层已经组装好的结构化数据
- `template_code`：执行器渲染时要用的模板
- `status`：如 `pending`、`sending`、`success`、`failed`

## 8. 执行器怎么工作

执行器建议做成独立服务，持续后台运行。

### 8.1 扫描

周期性扫描：

```sql
SELECT id
FROM aes_notify_batch
WHERE status = 'pending'
  AND next_retry_at <= NOW()
ORDER BY created_at
LIMIT 100;
```

### 8.2 抢占

对每条记录做抢占更新：

```sql
UPDATE aes_notify_batch
SET status = 'sending',
    updated_at = NOW()
WHERE id = ?
  AND status = 'pending';
```

只有抢占成功的 worker 才继续处理。

### 8.3 渲染

执行器根据：

- `template_code`
- `payload_snapshot`
- `channel_set`

生成企业微信、邮件、短信对应的发送内容。

### 8.4 发送

执行器调用 `notification` 二方库：

- 不建议执行器自己拼 HTTP
- 建议统一通过 client library 调底座

### 8.5 回写

发送成功：

- 更新 `success`

发送失败：

- 更新 `failed`
- 或有限次重试后再次回到 `pending`

## 9. 并发模型

执行器建议采用固定大小 worker pool。

例如：

- 扫描线程 1 个
- worker pool 并发数 `10`

这意味着：

- 不会无限并发压垮下游
- 也不会因为纯串行而吞吐太低

第一版固定并发 `10` 是一个比较合理的起点，后续可以根据压测结果调大或调小。

## 10. 多实例问题

无论是业务服务还是执行器，只要是多实例部署，就必须处理重复执行问题。

### 10.1 业务服务侧

如果某个业务服务有多个实例，不建议每个实例都跑相同的生产调度。

可以选择：

- leader election
- 分布式锁
- 数据库抢占

目标是：

- 一轮业务批次只能由一个实例负责生产

### 10.2 执行器侧

执行器可以多实例部署，但必须依赖数据库抢占，避免重复发送。

也就是：

- 多个实例都可以扫描
- 但只有抢占成功的实例才能真正发送

## 11. 为什么执行器不建议再暴露内部 HTTP API

如果执行器已经和业务服务通过批次表解耦了，就不建议再做一层“业务服务同步调执行器 HTTP API”。

原因是：

- 多一跳网络调用
- 多一个失败点
- 发送链路变成双重同步
- 故障处理更复杂

更合理的方式是：

- 业务服务写表
- 执行器扫表

这条链路更稳，也更适合多服务场景。

## 12. 推荐实现方式

结合当前诉求，推荐的第一版实现是：

- 编排层维护配置
- 各业务服务各自产批次
- 一个统一执行器扫表发送
- 执行器内部用 `notification` 二方库
- 固定并发 `10`
- 有限次重试

这版已经足够支撑：

- 多服务分散数据源
- 统一发送治理
- 下游限流保护
- 故障排查

## 13. 当前推荐结论

在多服务场景下，最合理的通知发送方案不是“每个服务各发各的”，而是：

- 生产分散在业务服务
- 发送集中到统一执行器

一句话总结：

多服务下，AES 执行层最稳的落地方式是“各业务服务产批次，公共执行器扫表发送”，这样既保留了业务侧的数据控制权，又把发送能力统一收口。
