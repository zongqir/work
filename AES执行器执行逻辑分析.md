# AES 执行器执行逻辑分析

## 1. 目标

这一部分不先讨论表结构细节，而是先把执行器本身的运行逻辑想清楚。

当前前提已经基本明确：

- 执行器由 `aes-crontab` 承载
- 业务微服务只负责写待发送批次
- 执行器统一扫描批次并发送
- 底座继续复用 `notification HTTP`

所以这一篇重点回答的是：

- 执行器到底怎么跑
- 什么时候扫
- 怎么避免重复发
- 怎么控制并发
- 怎么保护下游

## 2. 执行器的本质

执行器本质上不是“一个定时任务”，而是一套持续运行的后台处理循环。

它至少包含 4 个逻辑阶段：

1. 扫描待发送批次
2. 抢占批次
3. worker 执行发送
4. 回写结果

这 4 步是执行器最核心的骨架。

## 3. 扫描逻辑

执行器的第一步不是立刻发，而是先扫描“当前有哪些批次值得处理”。

筛选条件通常应包括：

- `status = pending`
- 已到可发送时间
- 未超过最大重试次数

这意味着扫描不是全表扫描，而是按条件挑出“当前该处理的那一批记录”。

### 扫描频率

第一版建议扫描频率不要太激进，也不要太慢。

例如：

- 每 `1s` 或 `3s` 扫一次

这类频率通常已经足够。

原因是：

- 批次不是毫秒级实时消息
- 但也不希望发送积压太久

### 分页扫描

每次扫描不建议无限取数，应该分页。

例如：

- 一次最多取 `100`
- 或一次最多取 `200`

这样可以避免：

- 一次性拖太多任务进内存
- 扫描和发送之间的节奏失衡

## 4. 抢占逻辑

扫描出来并不意味着立刻就能发。

如果执行器有多个 worker，甚至未来有多个实例，那就必须做抢占。

抢占的目的就是：

- 保证同一条批次同一时间只被一个 worker 处理

典型流程是：

1. 扫描出候选批次 ID
2. 对每条记录执行状态更新
3. 只有更新成功的 worker 才拥有处理权

也就是说，真正让记录进入发送流程的，不是“查出来了”，而是“抢占成功了”。

## 5. worker 执行模型

执行器不建议纯串行，也不建议无限并发。

当前更适合的方式是：

- 固定大小 worker pool

例如：

- worker 数量先设为 `10`

这意味着：

- 同时最多有 `10` 条批次在执行发送
- 超出的批次继续留在待处理队列中

这是一个很稳的折中方案。

## 6. worker 内部处理步骤

worker 拿到一条批次后，建议按下面顺序处理：

1. 读取批次内容
2. 校验批次字段是否合法
3. 根据 `template_code` 加载模板
4. 用 `payload` 渲染不同渠道内容
5. 构造 `notification` 请求
6. 调用 `notification client`
7. 根据结果更新状态

这里有个重要原则：

- worker 只处理“发送”
- worker 不负责回头查复杂业务数据

也就是说，业务层写进批次表的数据必须足够让执行器发出去。

## 7. 模板渲染逻辑

模板渲染应该发生在执行器里，而不是业务微服务里。

原因是：

- 企业微信、邮件、短信格式不同
- 渲染逻辑属于发送层通用能力
- 如果放回业务侧，会迅速分散

执行器里更合理的顺序是：

- 批次里带 `template_code`
- 批次里带结构化 `payload`
- 执行器负责渲染最终内容

## 8. 发送逻辑

发送逻辑建议统一通过 `notification client` 完成，而不是在执行器里到处直接拼 HTTP。

执行器应该只做：

- 构造标准化发送请求
- 调 `client.Send(...)`

不要把 HTTP 细节散落在多个 worker 处理函数里。

## 9. 发送成功后的处理

发送成功后，执行器需要至少做两件事：

1. 把状态改为 `success`
2. 清空错误信息或记录成功结果

如果底座会返回某个 `requestId` 或追踪信息，也建议一并记录下来，便于排障。

## 10. 发送失败后的处理

发送失败后，不建议立刻无限重试。

更合理的是：

- 记录本次错误
- 增加 `retry_count`
- 计算 `next_retry_at`
- 保持或回退为可重试状态

如果超过阈值：

- 最终停留为 `failed`

第一版建议把重试做轻：

- 最多 `1` 到 `3` 次
- 固定退避

这样复杂度可控。

## 11. 下游保护

执行器最重要的职责之一就是保护下游 `notification`。

这意味着即使批次很多，也不能无脑并发打出去。

建议至少控制以下几个维度：

- worker pool 固定并发
- 单次扫描数量上限
- 失败重试间隔
- 必要时按渠道做更细粒度节流

一句话说：

- 执行器不是单纯“快点发”
- 执行器更重要的是“稳地发”

## 12. 多实例执行器怎么处理

当前阶段，如果 `aes-crontab` 是多实例部署，我更建议做成：

- 分布式锁
- 单活执行器

也就是说：

- 多个实例都可以部署
- 但只有拿到分布式锁的实例进入执行循环
- 其他实例保持 standby

这样做的好处是：

- 第一版逻辑更简单
- 更容易排障
- 不需要一开始就处理多实例同时扫描带来的复杂度

### 12.1 锁的作用

分布式锁主要解决的是：

- 当前到底由哪个实例负责扫表和发送

也就是说，它控制的是：

- 谁是 active executor

### 12.2 锁不能替代状态推进

即使采用单活执行器，也仍然建议保留批次状态推进，例如：

- `pending`
- `sending`
- `success`
- `failed`

原因是：

- 执行器可能在处理中途崩溃
- 锁可能超时切换
- 某些发送记录需要恢复和补偿

所以分布式锁解决的是“实例级并发”，不是“任务级状态”。

### 12.3 锁需要具备的能力

如果采用分布式锁，至少要保证：

- 有过期时间
- 可以续约
- 实例挂掉后能自动释放
- 其他实例可以接管

否则会出现：

- 锁一直不释放
- 没有实例继续执行

### 12.4 为什么当前更适合单活

当前阶段更关注的是：

- 先把链路做稳
- 先把发送逻辑收拢
- 先把下游保护做好

在这个阶段，单活执行器通常比“多实例同时扫 + 竞争抢占”更合适。

后续如果发送量明显增大，再考虑演进成真正的多实例并发消费模型。

## 13. 执行器的真实运行形态

执行器不是“一次性跑完退出”，更像是一个持续后台循环。

可以理解成：

- 一个扫描线程
- 一个待处理队列
- 一个固定大小 worker pool
- 外加一个分布式锁控制当前 active executor

持续做下面这件事：

```text
scan -> claim -> enqueue -> render -> send -> update status
```

这个模型简单，但已经足够支撑第一版。

## 14. 当前推荐执行模型

结合当前讨论，我建议执行器第一版按这个模型落：

- `aes-crontab` 内部实现
- 通过分布式锁控制单活
- 每隔 `1s~3s` 扫一次
- 每次取固定数量批次
- 先抢占，再放进 worker pool
- worker pool 固定并发 `10`
- 执行器内部模板渲染
- 通过 `notification client` 发送
- 成功写 `success`
- 失败做有限次重试

## 15. 当前结论

执行器真正需要想清楚的，不是“有没有定时任务”，而是这几件事：

- 怎么扫
- 怎么抢
- 怎么并发
- 怎么限流
- 怎么回写

一句话总结：

AES 执行器本质上是一套“扫描 + 抢占 + worker pool + 发送 + 回写”的后台执行循环，而不是简单的一次性定时任务。
