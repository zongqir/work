# AES 通知二方库与执行器职责划分

## 1. 当前结论

在“纯通知执行层”定位下，工程落地建议这样划分：

- 上游业务通过一个 producer sdk 写入待发送批次
- `aes-crontab` 作为统一执行器扫描并发送
- 不再在 AES 里承担业务调度和业务汇总逻辑

一句话总结：

- 上游负责触发和组装请求
- AES 负责统一发送

## 2. 为什么只封装生产侧二方库

业务微服务很多，如果不统一封装写批次逻辑，后面很容易出现：

- 每个服务自己拼插入 SQL
- 每个服务自己定义请求字段
- 每个服务自己生成 `idempotency_key`
- 每个服务自己理解渠道和模板参数

这样协议会很快散掉。

因此，业务侧最值得抽成二方库的是：

- 通知请求写入能力

## 3. producer sdk 的职责

这层二方库建议只做轻封装，不承载业务调度和业务查询逻辑。

它负责：

- 定义标准化待发送批次结构
- 基础参数校验
- 生成 `idempotency_key`
- 封装 `insert batch`
- 支持 `upsert/insert ignore`

它不负责：

- 业务巡检
- 业务汇总
- 定时触发
- 模板渲染
- 调用 `notification`
- 扫描批次表

也就是说，它本质上是一个：

- `notify-producer-sdk`

## 4. 推荐的上游调用方式

上游业务完成自己的业务动作后，直接把“已决定要发”的消息写入批次表。

可以理解为：

```text
业务动作完成 -> 组装 notify request/value -> producer sdk insert batch
```

而不是：

```text
业务动作完成 -> 自己拼 SQL -> 自己写表
```

也不是：

```text
通知层去反查业务数据 -> 再决定要不要发
```

## 5. `aes-crontab` 为什么适合做执行器

执行器这部分的逻辑已经不是简单工具能力，而是完整服务逻辑：

- 扫描待发送批次
- 抢占
- 模板渲染
- 并发控制
- 限流
- 发送
- 回写状态

这部分更适合集中放在：

- `aes-crontab`

而不是再抽一个“执行器 SDK”。

## 6. `aes-crontab` 的职责

`aes-crontab` 作为执行器，建议直接承载以下能力：

- 周期扫描待发送批次表
- 抢占 `pending` 记录
- 加载模板
- 渲染企业微信、邮件、短信内容
- 固定 worker pool 并发发送
- 控制发送速率
- 调用 `notification`
- 更新 `success/failed`
- 对失败记录做有限次补发

这里要特别强调：

- 它负责的是发送执行
- 不是业务触发调度

## 7. 还需不需要单独封装 Notification Client

虽然执行器整体不建议再抽成二方库，但仍然建议保留一个轻量的 `notification client` 包。

原因是：

- 不要让 `aes-crontab` 到处散落 HTTP 调用细节
- 请求结构、超时、日志、错误处理可以统一
- 后续如果别的地方也要调 `notification`，还能复用

所以更准确的落地方式是：

- 不抽“执行器 SDK”
- 但保留“`notification HTTP client` 包”

## 8. 最终职责划分

### 8.1 上游业务微服务

负责：

- 业务动作本身
- 自己何时触发通知
- 组装 `value/payload`
- 调 producer sdk 写批次

### 8.2 producer sdk

负责：

- 标准化批次结构
- 幂等键生成
- 批次插入封装
- 基础校验

### 8.3 `aes-crontab`

负责：

- 扫描批次
- 抢占
- 读取配置
- 渲染
- 发送
- 回写状态

### 8.4 notification client

负责：

- 封装 `notification HTTP` 调用

## 9. 当前推荐结论

当前更务实的工程落地方式是：

- 上游业务自己决定何时产生通知请求
- 业务侧只抽一个“批次写入” producer sdk
- `aes-crontab` 直接做统一执行器
- 执行器内部依赖一个轻量 `notification client`

一句话总结：

真正值得封装成二方库的是“统一写批次”，真正值得集中承载的是 `aes-crontab` 里的发送执行逻辑；业务调度本身不属于 AES 通知层。
