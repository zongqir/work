# AES 共享 ConfigMap 动态生效设计

## 1. 目标

这一部分专门解决一个问题：

- 通知默认配置如果放在共享 `ConfigMap` 里，如何动态感知并生效

这里要特别关注的是通知执行域里的字段：

- `enabled`
- `channels`
- `receivers`
- `template_binding`
- `quota_rule`

## 2. 先给结论

共享 `ConfigMap` 可以动态生效，但要满足 4 个条件：

1. 不能继续使用 `subPath` 单文件挂载
2. 必须按目录挂载
3. 需要统一二方库做读取和解析
4. 需要通过“快照重建 + 快照对比”更新执行层配置视图

一句话总结：

- `ConfigMap` 自己不会帮你完成业务生效
- 真正生效依赖挂载方式、加载方式和快照切换机制

## 3. 当前挂载方式为什么不行

结合当前仓库里的 Helm 模板，像 `aes-crontab` 和 `aes-console` 现在都是：

- 挂到 `/aes/config/config.yaml`
- 使用 `subPath: config.yaml`

这种方式的问题是：

- `ConfigMap` 更新后，容器内文件不会像目录挂载那样自动刷新

所以如果通知默认配置继续这么挂：

- 动态感知基本做不到

## 4. 推荐的 K8s 挂载方式

更合理的方式是：

- 单独一个共享 `ConfigMap`
- 各服务按目录挂载
- 不使用 `subPath`

例如：

- `ConfigMap` 名称：`aes-notify-defaults`
- 容器内目录：`/aes/config/notify-defaults/`
- 文件名：`notify-defaults.yaml`

这样挂载后：

- K8s 更新 `ConfigMap`
- 目录里的文件会被更新
- 应用侧才有机会 reload

## 5. 推荐的 K8s 编排方式

### 5.1 独立 Helm 包

推荐单独做一个 Helm 包，只负责创建共享通知默认配置：

- `aes-notify-defaults`

### 5.2 各服务单独挂载

需要读取通知默认配置的服务，在自己的 Deployment 中增加：

- `volume`
- `volumeMount`

统一挂载目录，例如：

- `/aes/config/notify-defaults/`

### 5.3 哪些服务需要挂

至少包括：

- `aes-crontab`
- 负责配置展示的控制面服务

## 6. 二方库负责什么

共享 `ConfigMap` 允许多服务消费后，就不能让每个服务自己 parse 一套。

应该有统一二方库负责：

- 读取 `notify-defaults.yaml`
- 解析统一 schema
- 基础校验
- 周期性 reload
- 输出结构化 `common` 配置

它不负责：

- 业务调度
- 业务汇总
- 业务数据查询

它只是：

- 默认配置加载器

## 7. 推荐的运行时模型

当前更推荐统一成：

1. 周期性 reload `ConfigMap`
2. 周期性 reload DB 中的租户配置
3. 构建新的稳定快照
4. 与旧快照比较
5. 不同则原子替换

第一版建议：

- 每 `60s` 执行一次

## 8. 为什么不建议单独维护 version

当前不推荐为了动态生效再引入：

- `default_config_version`

原因是：

- 快照本体已经是稳定结构
- reload 后直接比较新旧快照内容就够了

## 9. 快照变化后意味着什么

在纯通知执行层定位下，快照变化只意味着：

- 后续新的发送动作应按新配置执行

它不再意味着：

- 重算业务调度
- 重算业务窗口
- 重算业务 checkpoint

所以动态生效的语义明显更简单。

## 10. 推荐的整体流程

### 10.1 K8s 层

1. 独立 Helm 包发布 `aes-notify-defaults`
2. 相关服务按目录挂载配置目录

### 10.2 代码层

1. 二方库读取 `notify-defaults.yaml`
2. 周期性 reload
3. 同时读取 DB 中租户特殊配置
4. 构建新的稳定快照
5. 比较新旧快照并原子替换

### 10.3 执行层

1. 执行器读取当前快照
2. 新的批次按新配置执行
3. 已落表批次按冻结参数执行

## 11. 不推荐的方式

### 11.1 不推荐继续用 `subPath`

因为动态更新基本不成立。

### 11.2 不推荐每个服务自己 parse

因为 schema 和语义会散。

### 11.3 不推荐配置变化后回头改老批次

因为会让已落表批次的执行行为漂移。

## 12. 当前推荐结论

共享通知默认配置的动态生效，当前最合理的设计是：

- 独立公共 `ConfigMap`
- 目录挂载
- 统一二方库周期性 reload
- 构建稳定快照
- 用快照对比识别变化
- 后续新的发送动作按新配置执行

一句话总结：

`ConfigMap` 动态生效不是单靠 Kubernetes 完成的，它本质上是“目录挂载 + 统一解析 + 稳定快照 + 原子切换”这一整套链路。
