# 服务端可靠推送方案整理

本文整理了这次关于“服务端可靠推送”讨论过的方案、边界、优缺点和推荐落地路径。这里的“可靠”默认不是绝对送达，而是工程上可验证、可补偿、可重试的最终一致送达。

## 1. 先统一几个结论

服务端推送里，通常要分清 4 个层次：

- 已创建：业务侧生成了一条待推送消息
- 已投递到推送通道：服务端成功调用了 Kafka / APNs / FCM / WebSocket / SSE / 厂商通道
- 设备已收到：客户端连接层或 SDK 确认收到了
- 用户已处理：客户端展示、打开、点击、回 ACK

大多数团队真正做的是：

- 发送链路做到 `at-least-once`
- 客户端消费做到幂等
- 用 ACK、状态机、重试、补偿兜底
- 接受“Push 通道本身不是强保证”这个现实

移动端尤其如此：

- 在线时优先走长连接或实时通道
- 离线时走系统 Push
- 恢复在线后主动拉未读
- 服务端以“ACK 或客户端拉取成功”为最终送达依据

## 2. 方案总览

这次讨论过的方案主要有：

1. 纯定时任务补偿
2. 扫业务表
3. 扫业务表 + `push_record`
4. Outbox Pattern
5. CDC / Binlog 订阅
6. 事务消息
7. 纯 MQ 驱动
8. 推拉结合
9. 工作流 / 任务平台编排

下面分别说明。

## 3. 方案一：纯定时任务补偿

### 核心思路

- 主流程先尽力发一次
- 定时任务不断扫描“未成功”或“未 ACK”的消息
- 到时间就重试
- 超过阈值标记失败或进入死信

### 典型字段

- `message_id`
- `user_id`
- `payload`
- `status`
- `retry_count`
- `next_retry_at`
- `max_retry`
- `ack_at`
- `last_error`

### 典型状态

- `pending`
- `sending`
- `sent`
- `acked`
- `failed`

### 优点

- 简单
- 好落地
- 对“最终一致送达”很实用

### 缺点

- 实时性一般
- 必须接受重复推送
- 不能只靠定时任务证明送达，最好仍然有 ACK

### 适用场景

- 通知类、提醒类、补偿类业务
- 可以接受秒级到分钟级延迟

## 4. 方案二：扫业务表

### 核心思路

- 定时任务扫描业务表
- 找出“当前满足推送条件”的业务数据
- 对这些数据发推送
- 下次继续扫

### 典型做法

不要只按“上次推送时间”过滤，而要用稳定游标：

- `(updated_at, id)`
- `(version, id)`
- 或纯递增 `id`

推荐查询方式：

```sql
SELECT *
FROM biz_table
WHERE (updated_at > :last_updated_at)
   OR (updated_at = :last_updated_at AND id > :last_id)
ORDER BY updated_at, id
LIMIT 1000;
```

扫描完成后，把这批里最后一条的 `(updated_at, id)` 作为新游标。

### 不建议只用“上次推送时间”的原因

- 多条数据时间戳相同，容易漏
- `>` 和 `>=` 很容易写错边界
- 扫描过程中有新数据写入时，边界不稳定
- 如果把游标直接更新成 `now()`，中间写入的数据可能被跳过

### 优点

- 最简单
- 对通知、补发、周期提醒很实用

### 缺点

- 适合“按当前状态补发”
- 不适合“某次状态变化必须推一次”的强事件语义

### 适用场景

- 订单超时提醒
- 待审批提醒
- 长时间未处理工单提醒

## 5. 方案三：扫业务表 + `push_record`

这是当前阶段最推荐的折中方案。

### 核心思路

- 仍然按业务表做增量扫描
- 但不是扫到就直接推
- 先写一张 `push_record`
- 对 `biz_id + push_type` 或 `message_id` 做唯一约束
- 写成功后再发推送
- 收到 ACK 后停止重试

### 典型表设计

业务扫描游标表：

- `job_name`
- `last_updated_at`
- `last_id`
- `updated_at`

推送记录表：

- `id`
- `biz_id`
- `push_type`
- `message_id`
- `status`
- `retry_count`
- `next_retry_at`
- `provider_msg_id`
- `last_error`
- `acked_at`
- `created_at`
- `updated_at`

### 建议顺序

1. 按游标扫描业务表
2. 对候选业务写 `push_record`
3. 写成功后发推送
4. 推送成功后更新 `push_record`
5. 最后推进扫描游标

### 关键原则

- 绝不能先推进游标再推送
- 推送一定要幂等
- ACK 是最终停止重试的依据之一
- 可以做“重叠扫描窗口”，例如每次回退 1 到 5 分钟，再靠幂等去重，提升防漏能力

### 优点

- 比 Outbox 简单很多
- 比纯扫业务表稳很多
- 有幂等、审计、补偿能力

### 缺点

- 仍然不是强事件语义
- 需要维护一张额外的推送记录表

### 适用场景

- 大多数业务通知
- 第一版可靠推送链路
- 不想一开始就上 Outbox 的场景

## 6. 方案四：Outbox Pattern

中文常叫：

- 事务外盒
- 本地消息表
- 投递箱模式

### 核心思路

在同一个数据库事务里，同时写：

- 业务数据
- 一条待投递的 `outbox` 记录

事务提交后，由后台投递器异步把 `outbox` 记录发送到 Kafka / MQ / Push 服务。

### 正确时序

不是：

1. 业务 + outbox 放事务里
2. 事务里直接发 Kafka
3. 再在事务里更新 outbox

而是：

1. 本地数据库事务里写业务数据和 `outbox`
2. 事务提交成功
3. 后台 `relay/poller` 扫描 `outbox`
4. 发 Kafka / MQ / Push
5. 发送成功后更新 `outbox.status = sent`

### 优点

- 解决“业务提交成功，但消息没发出去”的一致性问题
- 关键业务事件不容易漏

### 缺点

- 相对更重
- 需要维护 `outbox` 状态、投递器、重试和补偿
- 仍然要接受重复投递，所以消费者还得幂等

### 典型场景

- 支付成功
- 风控告警
- 强业务事件通知
- 跨系统联动不能漏的场景

### 本质说明

Outbox 保证的是：

- 消息一定先被可靠记录下来

它不保证：

- 下游绝对只收到一次

所以 Outbox 不是 exactly-once，而是更稳的事件持久化起点。

## 7. 方案五：CDC / Binlog 订阅

### 核心思路

- 直接订阅数据库变更日志
- 把业务表变更流式同步到下游
- 再由下游进行推送或进一步加工

### 优点

- 不需要频繁扫表
- 适合数据量较大场景
- 以数据变更为事实源

### 缺点

- 运维和链路理解门槛更高
- 仍要处理幂等、过滤和状态识别

### 适用场景

- 团队已经有成熟的 CDC 基础设施
- 数据变更天然就是业务事件

## 8. 方案六：事务消息

### 核心思路

- 利用支持事务消息的 MQ 机制
- 业务本地事务和消息发送通过 MQ 的事务能力联动
- 必要时由 MQ 回查本地事务状态

### 优点

- 某些中间件下，一致性体验更自然
- 不一定要自己维护完整 Outbox 流程

### 缺点

- 强依赖具体 MQ 能力
- 不是所有中间件都适合这条路
- Kafka 原生不是典型事务消息方案

### 适用场景

- 团队本来就深度依赖支持事务消息的 MQ

## 9. 方案七：纯 MQ 驱动

### 核心思路

- 业务成功后直接发 MQ
- 推送服务消费 MQ
- 消费端做重试、死信、ACK

### 优点

- 架构清晰
- 实时性较好

### 缺点

- 如果业务成功了，但发 MQ 失败，中间可能漏

- 可以接受少量上游一致性风险
- 或者已有补偿流程兜底

## 10. 方案八：推拉结合

这是移动端和在线系统里非常常见的可靠思路。

### 核心思路

- 服务端先推送提醒
- 客户端在线时实时接收
- 客户端恢复在线后主动拉未读
- 服务端以“ACK 或拉取成功”作为送达依据

### 优点

- 不完全依赖系统 Push
- 在 App 离线、厂商通道不稳定时更稳
- 更接近真正的最终送达

### 缺点

- 客户端和服务端都要配合
- 协议和状态机会多一些

### 适用场景

- App Push
- WebSocket / SSE
- 站内信 + Push 结合

## 11. 方案九：工作流 / 任务平台编排

### 核心思路

- 把推送当成一类任务或工作流
- 支持超时、重试、分支、人工补偿、升级通知

### 优点

- 复杂业务表达能力强
- 适合多步骤通知链路

### 缺点

- 明显更重
- 对“只是发条通知”的场景可能过度设计

### 适用场景

- 推送只是业务流程的一环
- 有人工处理、短信补偿、电话补偿等复杂升级路径

## 12. 关于“按上次推送时间扫描业务”的判断

讨论过一个简化思路：

- 先改业务
- 再通过一个配置关注业务状态
- 定时任务检查“上次推送时间”
- 按这个时间去筛选需要推送的业务
- 推送
- 更新上次推送时间
- 如果更新失败，下次继续

这个思路本质上是：

- 增量扫描
- 水位推进
- 最终一致补偿

它能做，但要注意：

- 不建议只存“上次推送时间”
- 最好用稳定游标，例如 `(updated_at, id)`
- 游标推进必须在“本批处理完成后”
- 即使游标更新失败，下次能继续的前提也是“推送链路幂等”

真正危险的是：

- 游标已经推进了
- 但推送实际上没完成或没记账

这会导致漏推。

所以顺序必须是：

1. 扫描候选业务
2. 记推送记录
3. 发推送
4. 成功后更新状态
5. 最后推进扫描游标

## 13. 方案复杂度和推荐顺序

按复杂度从低到高，大致可以理解为：

1. 扫业务表
2. 扫业务表 + `push_record`
3. 纯 MQ 驱动
4. Outbox
5. CDC
6. 事务消息
7. 工作流 / 任务平台

按“对关键事件防漏能力”从弱到强，大致可以理解为：

1. 扫业务表
2. 扫业务表 + `push_record`
3. 纯 MQ 驱动
4. Outbox
5. 事务消息
6. CDC
7. 推拉结合

这里没有绝对排名，实际还要看：

- 业务语义是“当前状态提醒”还是“状态变化事件”
- 是否要求严格实时
- 是否允许重复
- 是否允许分钟级补偿
- 团队是否具备 MQ / CDC / 工作流平台基础设施

## 14. 当前最推荐的落地路径

如果目标是：

- 先把可靠推送做起来
- 不想一上来搞太重
- 可以接受最终一致

推荐优先做：

### 第一阶段

- 扫业务表
- 用 `(updated_at, id)` 作为游标
- 增加 `push_record`
- 对 `biz_id + push_type` 或 `message_id` 做唯一约束
- 推送后等 ACK
- ACK 前按定时任务补偿重试

### 第二阶段

只把少数“关键业务事件”升级为 Outbox：

- 支付成功
- 风控告警
- 核心异步通知

### 第三阶段

如果是移动端或在线系统，再补：

- 客户端拉未读
- ACK 回执
- 推拉结合

## 15. 一句话选型建议

- 想要简单够用：`扫业务表 + push_record + ACK + 定时补偿`
- 想要关键事件不漏：`Outbox`
- 已有强中间件能力：`事务消息` 或 `CDC`
- 想做真正最终送达：`推拉结合`

## 16. 最后总结

这次讨论里最重要的判断是：

- 不要追求“绝对不丢、绝对不重”的幻觉
- 重点是让链路变成“可记录、可重试、可幂等、可补偿、可确认”
- 如果当前阶段优先级是低复杂度落地，不必一开始就上 Outbox
- 第一版最合适的通常是：增量扫描业务表 + `push_record` + ACK + 定时补偿

这套方案通常已经足够支撑大多数通知类和提醒类推送场景。
