# DB 调度方案与 MQ 方案对比

## 1. 背景

当前通知方案里，一个关键设计点是：

- 是否需要引入 MQ

结合当前讨论，当前更偏向的方案是：

- 不额外引入 MQ
- 使用数据库表 + 高频调度扫描 + 执行器发送

这意味着，本质上是在用 DB 调度方案替代一部分 MQ 的作用。

## 2. 当前方案是什么

当前更落地的方案可以概括为：

- 策略配置放 `configmap + override 表`
- 业务服务产待发送批次
- 批次写入数据库表
- 调度器高频扫描
- 执行器读取并发送

一句话理解：

- 用数据库表承载待处理任务
- 用调度器轮询代替消息消费

## 3. MQ 方案通常解决什么

如果采用 MQ，通常会获得这些能力：

- 异步解耦
- 消息缓冲
- 任务分发
- 削峰填谷
- 消费者并发扩展

通知链路里，MQ 经常用来承接：

- 业务侧投递任务
- 执行侧消费任务

## 4. 当前 DB 调度方案替代了 MQ 的哪些作用

虽然没有 MQ，但当前方案已经在用别的方式承担类似角色。

### 4.1 批次表

批次表承担的是：

- 待消费消息队列

也就是说，业务服务把任务写进去，执行器后续再消费。

### 4.2 调度状态表

调度状态表承担的是：

- 任务触发时机管理

它决定：

- 什么时候该生产批次
- 什么时候该执行

### 4.3 执行器

执行器承担的是：

- 消费者
- worker

它负责：

- 扫描
- 抢占
- 发送
- 回写

一句话总结：

当前方案虽然没有 MQ，但已经用：

- 表 + 调度器 + worker

替代了：

- 队列 + consumer

## 5. DB 调度方案的优点

### 5.1 简单

这是当前方案最大的优点。

- 不需要新增 MQ 基础设施
- 不需要额外处理 topic、consumer group、消息格式
- 更容易快速落地

### 5.2 易查

数据库表天然适合排障。

可以直接看：

- 哪条任务没发
- 哪条任务失败
- 什么时候该发
- 为什么没发

### 5.3 更符合当前阶段

当前前提是：

- 不想把系统做太重
- 通知链路第一版更看重可控和可落地
- 底座也不明显倾向于做重型 MQ 通知能力

所以 DB 调度方案更符合现状。

## 6. DB 调度方案的代价

### 6.1 实时性取决于扫描频率

没有 MQ 的即时投递后，系统只能靠：

- 高频扫描

所以执行精度取决于：

- 扫描间隔

### 6.2 数据库会承担更多角色

数据库不仅存业务数据，还要承担：

- 调度状态
- 待发送批次
- 重试状态

这会增加数据库压力。

### 6.3 吞吐上限不如 MQ

如果未来批量任务很多、发送量很大，DB 调度方案的扩展性通常不如 MQ。

## 7. MQ 方案的优点

### 7.1 天然异步

业务侧投递后即可返回，后续消费交给 MQ。

### 7.2 更适合高吞吐

大规模消息分发时，MQ 的扩展性通常更强。

### 7.3 消费模型更成熟

很多成熟能力在 MQ 里天然存在：

- consumer group
- backlog
- 死信
- 重试

## 8. MQ 方案的代价

### 8.1 更重

需要：

- 部署和维护 MQ
- 定义 topic
- 管理消费组
- 处理消息顺序、积压、重试策略

### 8.2 第一版复杂度更高

相比 DB 调度，MQ 方案初期投入明显更大。

### 8.3 与当前底座方向不完全一致

当前已知前提里，底座并不明显倾向于专门给一套更重的 MQ 通知能力。

因此直接上 MQ 可能和当前建设倾向不一致。

## 9. 当前阶段更推荐哪种

结合当前已知条件：

- 多微服务
- 不想一开始做太重
- 现有底座偏 HTTP
- 已有 OceanBase
- 希望快速收敛方案

当前更推荐：

- DB 调度方案

也就是：

- 不额外引入 MQ
- 用数据库表和调度器承担异步解耦

## 10. 可以怎么表述当前方案

可以把当前方案明确描述为：

- 不是基于 MQ 的通知调度架构
- 而是基于数据库表的轻量异步调度架构

这套架构的核心就是：

- 用高频调度扫描代替 MQ 消费
- 用批次表代替消息队列

## 11. 后续演进方向

如果后续出现这些情况：

- 批次量明显增大
- 调度扫描成本过高
- 对实时性要求明显提高
- 对削峰和解耦要求更强

那时再评估演进到 MQ 更合适。

也就是说，当前并不是否定 MQ，而是：

- 先不急着上 MQ

## 12. 当前结论

当前方案确实省掉了一层 MQ。

但这不是“少做了一层”，而是：

- 用数据库表、调度器和执行器把 MQ 的一部分作用接住了

一句话总结：

当前更务实的方案是“用 DB 调度替代 MQ”，先把通知链路做成一套轻量异步系统；如果未来规模和实时性要求继续上升，再考虑演进成 MQ 方案。
