# AES 双通道执行器调度设计

## 1. 背景

在当前通知场景中，待发送任务并不是同一种类型。

至少可以分成两类：

- 响应式任务
- 后台批量任务

这两类任务的目标完全不同：

- 响应式任务强调低延迟
- 后台批量任务强调吞吐和稳定

如果把它们完全混在同一条执行通道里，后台批量任务很容易把响应式任务挤住。

## 2. 核心结论

如果明确要求“响应式通知不能被后台任务阻塞”，那么执行器就不应该只有一条任务通道。

更合理的方案是：

- 两张表
- 两条执行通道

一句话总结：

- 响应式走响应式通道
- 后台任务走后台通道

## 3. 推荐拆分方式

### 3.1 响应式批次表

建议表名：

- `aes_notify_realtime_batch`

适合承载：

- 安全事件即时通知
- 新安全告警通知
- 管理员登录通知
- 任务状态变更通知

特点：

- 低延迟
- 单条或小批量
- 希望尽快发送

### 3.2 后台批次表

建议表名：

- `aes_notify_schedule_batch`

适合承载：

- 每天固定时间汇总任务
- 每 `10` 分钟/每小时汇总任务
- 阈值扫描任务
- 周期性通知任务

特点：

- 批量
- 可接受一定排队
- 更偏吞吐

## 4. 为什么不建议只用一张表加 priority

理论上，也可以只用一张表，再加：

- `priority`

但当前阶段我不建议这么做，原因是：

- 扫描逻辑还是混在一起
- 抢占逻辑还是混在一起
- worker pool 还是共用
- 后台大批量任务依然可能占满执行资源

也就是说，`priority` 能表达优先级，但不能天然提供执行隔离。

如果你真正想解决的是：

- 响应式任务不能被后台阻塞

那么只靠优先级通常不够。

## 5. 推荐执行模型

虽然建议两张表，但不建议做成两个完全独立服务。

更合理的方式是：

- 一个 `aes-crontab` 执行器进程
- 两个扫描器
- 两个 worker pool
- 一个共享 `notification client`

也就是说，执行逻辑集中，但通道隔离。

## 6. 响应式通道

响应式通道建议：

- 扫描更频繁
- 每次取数更小
- worker pool 独立
- 更强调低延迟

例如：

- 每 `1s` 扫一次
- 每次抓 `20`
- worker pool 并发 `5`

这条通道的目标不是高吞吐，而是快速处理。

## 7. 后台通道

后台通道建议：

- 扫描相对慢一点
- 每次取数更大
- worker pool 独立
- 更强调吞吐和稳定

例如：

- 每 `3s` 或 `5s` 扫一次
- 每次抓 `100`
- worker pool 并发 `10`

这条通道可以慢一些，但不能无限积压。

## 8. 为什么两条通道更稳

两条通道的价值在于：

- 响应式任务有自己的执行资源
- 后台任务有自己的执行资源
- 两边互相影响更小

例如：

- 即使后台批量任务突然堆积很多
- 响应式通道仍然有自己的 worker 可以发消息

这就是物理隔离带来的稳定性。

## 9. 下游保护还要继续做

即使拆成两条通道，也不能让它们一起把 `notification` 打爆。

因此还建议继续做：

- 通道级并发控制
- 通道级节流
- 全局发送上限

更进一步一点，可以考虑：

- 响应式通道保底额度
- 后台通道保底额度

这样可以避免：

- 某一边完全抢光底座发送能力

## 10. 状态设计

两张表虽然分开，但状态模型建议保持一致。

例如都使用：

- `pending`
- `sending`
- `success`
- `failed`

这样有几个好处：

- 执行逻辑可复用
- 状态语义统一
- 监控和排障更容易做

## 11. 多实例执行器怎么配合

当前阶段既然已经倾向于：

- 分布式锁
- 单活执行器

那么两条通道也可以共用这一套模型。

也就是说：

- 当前只有一个 active executor
- 这个 active executor 同时运行响应式扫描器和后台扫描器
- 其他实例 standby

这样实现简单，行为也更容易理解。

## 12. 推荐的执行流程

### 12.1 响应式流程

1. 业务服务写入 `aes_notify_realtime_batch`
2. 响应式扫描器扫出 `pending`
3. 抢占成 `sending`
4. 放入响应式 worker pool
5. 调 `notification`
6. 回写 `success/failed`

### 12.2 后台流程

1. 业务服务写入 `aes_notify_schedule_batch`
2. 后台扫描器扫出 `pending`
3. 抢占成 `sending`
4. 放入后台 worker pool
5. 调 `notification`
6. 回写 `success/failed`

## 13. 当前推荐结论

当前阶段，如果明确要求响应式通知不能被后台任务阻塞，最务实的设计就是：

- 两张表
- 两个扫描器
- 两个 worker pool
- 一个执行器进程
- 一个统一 `notification client`

一句话总结：

双通道执行器的核心不是“多加几条 if 分支”，而是把响应式任务和后台任务在执行资源上真正隔离开，这样才能保证响应式通知不被批量任务拖住。
