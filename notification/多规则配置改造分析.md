# 多规则通知改造分析

## 背景

当前通知系统的核心假设是：

- 一个 `tenant + message_type` 只有一套生效配置
- 这套配置里只有一组 `filter`
- 这套配置里只有一个 `channel`
- 聚合调度水位只按 `tenant + message_type` 维护

你现在描述的新诉求是：

- 同一个 `tenant`
- 同一个 `message_type`
- 可能存在多套条件
- 每套条件的实时/聚合开关、聚合周期、接收人、渠道都可能不同

但这里有一个重要变化：

1. 你不再是配置提供方，而是调用别人的接口拿配置
2. 你可以在本系统前面加一层“配置转换器”，把外部配置打平成内部格式
3. 你在考虑在 `handler` 外面再包一层，让它能处理一个 `message_type` 下的多条配置

这个变化会显著降低改造复杂度，因为最难的部分不是“配置来源是谁”，而是“系统内部是不是仍然坚持单配置模型”。

## 结论

结论先说：

- 如果你愿意在接入层把外部多规则配置打平成“内部规则列表”，这次改造不是特别大
- 但它也不是只包一层 `handler` 就结束
- 真正合理的切点不是“包 handler”，而是“在 dispatcher 层引入 rule 维度”

换句话说：

- “配置来源外部化”确实让问题简单了
- “接入层做转换”这个方向是对的
- “只在 handler 外包一层 list 处理”这个方案不够完整，我不建议把它作为主方案

## 为什么你觉得没那么复杂，这个判断有道理

你这个判断成立的前提是：外部配置系统已经帮你解决了大部分管理复杂度。

你现在不需要在本仓库里解决这些问题：

- 配置录入
- 配置存储模型设计
- 配置编辑 UI
- 配置版本管理
- 多规则的管理台校验

这意味着本系统只需要解决：

- 如何加载外部配置
- 如何转换成内部可执行规则
- 如何在分发、聚合、投递链路上按规则执行

从这个角度说，复杂度确实比“从零把单配置模型升级成完整规则引擎”低很多。

## 你提出的三点思路，我的判断

### 1. 不再是配置提供方，只调别人接口

这是利好，不是风险。

它直接减少了本系统的职责：

- 你不需要改 DAO 和默认配置文件去承载复杂规则模型
- 你不需要维护一套本地 schema 作为配置事实来源
- `config/cache` 仍然可以保留，缓存的只是“外部接口返回后经过转换的内部规则”

这里真正要注意的不是数据来源，而是内部缓存的结果长什么样。

我建议把现在的：

`tenant -> message_type -> raw_config`

改成：

`tenant -> message_type -> []rule`

这里的 `rule` 是你们内部运行时结构，不一定需要和外部接口完全一致。

### 2. 在转换器层把配置打平

这个方向我认可，而且我认为这是本次改造的关键。

“打平”后的目标不是继续产出当前这一个 `MessageConfig`，而是产出一组内部规则，例如：

```json
[
  {
    "rule_id": "r1",
    "tenant_id": "t1",
    "message_type": "risk_digest",
    "realtime_enabled": true,
    "aggregate_enabled": false,
    "filter": { "severity": ["high"] },
    "channel": {
      "channel": "email",
      "template_code": "risk_digest_default",
      "audience": { "to": ["a@x.com"] }
    }
  },
  {
    "rule_id": "r2",
    "tenant_id": "t1",
    "message_type": "risk_digest",
    "realtime_enabled": false,
    "aggregate_enabled": true,
    "aggregate_period_minutes": 60,
    "filter": { "severity": ["critical"] },
    "channel": {
      "channel": "wecom",
      "template_code": "risk_digest_default",
      "audience": { "phone": ["13800000000"] },
      "delivery": { "secret": "...", "agent_id": "..." }
    }
  }
]
```

这个“打平”设计有两个明显好处：

1. 外部接口再复杂，也只在转换层消化一次
2. 后面的 dispatcher / scheduler / processor 都只面对统一规则结构，不需要知道外部配置长什么样

这一步我基本没有反对意见。

### 3. 在 handler 外包一层，处理 list 配置

这个思路我部分同意，但我不建议把“多规则处理”压到 handler 外面。

原因是 `handler` 当前职责本来就不是“规则路由器”，而是“业务判断器”：

- `Evaluate(...)` 负责判断单次实时事件是否命中
- `Aggregate(...)` 负责按时间窗口聚合
- `NewFilter()` 负责提供 filter 结构

如果你把多规则逻辑包在 handler 外面，会碰到三个问题。

#### 问题一：实时和聚合的路由责任不在 handler

当前决定“要不要跑实时/聚合”的入口在 `Dispatcher`，不是 handler。

也就是说，当前链路是：

1. `Dispatcher` 取配置
2. `Dispatcher` 判断 `realtime_enabled / aggregate_enabled`
3. `Dispatcher` 调用 handler
4. `Dispatcher` 发布消息

所以多规则最自然的地方应该是：

1. `Dispatcher` 取规则列表
2. `Dispatcher` 遍历规则
3. 对每条规则分别判断实时/聚合是否开启
4. 对每条规则分别调用同一个 handler
5. 每条命中的规则各自产生一条 dispatch message

如果把 list 逻辑塞到 handler 包装层，就会出现职责反转：

- dispatcher 还以为自己面对的是单配置
- 但实际上 handler wrapper 已经在偷偷做多规则分发

这个抽象会变得别扭。

#### 问题二：消费侧最终还是需要知道“命中了哪条规则”

就算 wrapper 在发送前解决了“多规则匹配”，到消费投递阶段仍然有问题。

当前 `DispatchMessage` 只有：

- `tenant_id`
- `message_type`
- `source`
- `biz_vars`

没有：

- `rule_id`

这会带来两个风险：

1. 投递侧不知道该按哪条规则取渠道和接收人
2. 如果消息入队后配置发生变化，消费时会查到错误的目标配置

所以即使你在 handler 外面包一层，`DispatchMessage` 最终还是需要带上规则身份。

#### 问题三：聚合调度绕不过 rule 维度

实时链路还可以勉强由 wrapper 帮忙“循环调用”。

但聚合链路不行，因为聚合还有调度水位问题：

- 规则 A 每 5 分钟聚合
- 规则 B 每 60 分钟聚合
- 它们同属一个 `tenant + message_type`

如果水位仍然只有一份，就会互相覆盖。

所以调度器必须知道：

- 当前有哪些规则启用了聚合
- 每条规则的聚合周期是多少
- 每条规则自己的 `last_window_end` 是多少

这件事不适合放在 handler wrapper 里做。

## 我对方案的反驳

如果你的方案被表述为：

“我在配置转换器里把它打平，然后只在 handler 外包一层 list handler，其他地方尽量不动。”

那我的反驳是：

这个方案只解决了“多条规则怎么跑业务判断”，没有完整解决下面三件事：

- 消息发布后的规则身份保持
- 投递侧如何稳定找到正确接收人/渠道
- 聚合调度如何按规则独立推进水位

因此它不是一个闭环方案。

## 我建议的更稳妥方案

### 核心原则

不要把系统升级成“复杂规则引擎”。

只做一件事：

把当前“单配置分发”升级成“规则列表分发”。

### 最小改造思路

保留现有 handler 契约不动：

- `NewFilter()`
- `Evaluate(...)`
- `Aggregate(...)`

新增一个平台内部运行时结构，例如：

```go
type MessageRule struct {
    RuleID                 string
    TenantID               string
    MessageType            string
    RealtimeEnabled        bool
    AggregateEnabled       bool
    AggregatePeriodMinutes int
    Filter                 json.RawMessage
    Channel                render.ChannelPolicy
}
```

然后改 4 个位置。

#### 1. 配置加载层

把外部接口返回内容转换成：

`tenant + message_type -> []MessageRule`

这里是最适合放“打平转换器”的地方。

#### 2. Dispatcher

把当前“加载一条配置”改成“加载规则列表”。

实时：

- 遍历所有 `RealtimeEnabled` 的规则
- 逐条解析 filter
- 逐条调用 `Evaluate(...)`
- 命中的规则各发一条消息

聚合：

- `SendAggregate(...)` 增加 `rule` 或 `rule_id` 维度
- 聚合消息按规则分别触发

#### 3. DispatchMessage

补一个 `RuleID` 字段。

这样消费侧才能稳定定位规则，并且幂等 key 也应带上 `rule_id`。

例如：

- 实时：`realtime:{tenant}:{message_type}:{rule_id}:{biz_key}`
- 聚合：`aggregate:{tenant}:{message_type}:{rule_id}:{window_start}:{window_end}`

#### 4. Scheduler

聚合水位升级为：

`tenant + message_type + rule_id`

否则多条聚合规则一定互相打架。

## 改动量评估

如果按上面的“最小改造思路”做，我认为改动量是：

### 不大的部分

- handler 契约基本不用动
- 渲染和 sender 基本不用大动
- 外部接口复杂度可以收敛在转换器

### 中等改动的部分

- 配置缓存结构
- dispatcher 逻辑
- dispatch message 结构
- processor 按 `rule_id` 回查规则
- scheduler 水位维度

### 不建议硬保留的部分

- “系统内部仍只有一个 MessageConfig，只在 handler 外包 list”

这会把复杂度藏起来，不会真的消失。

## 复杂度真实落点

如果只看“调用业务 handler 做判断”这一步，确实没那么复杂。

但真正的复杂度不在这里，而在下面三处：

1. 一条消息命中后，如何在异步链路里保留规则身份
2. 同一 `message_type` 多条聚合规则如何独立调度
3. 配置变化后，如何保证消费侧不会取错规则

所以我会这样定性：

- 不是大改架构
- 不是上规则引擎
- 但也不是简单包一层 wrapper 就能结束
- 正确级别是“中等规模的链路升级”

## 推荐落地方案

我建议按下面顺序做。

### Phase 1

先把外部配置转换成内部 `[]rule`，但先只支持实时。

目标：

- 验证一条事件命中多规则的行为
- 验证 `rule_id` 透传是否稳定
- 验证 processor 是否能按 `rule_id` 找到正确接收人

### Phase 2

再把聚合调度升级到 rule 维度。

目标：

- 每条 rule 独立周期
- 每条 rule 独立水位
- 聚合幂等键带 `rule_id`

### Phase 3

最后再看是否需要补高级能力：

- 规则优先级
- 首条命中即停止
- 规则互斥
- 命中日志与可观测性

这些都不应该在第一版就做。

## 最终判断

你的直觉有一半是对的。

对的部分：

- 外部配置接口 + 本地打平，确实把问题降维了
- 没必要上很重的抽象
- handler 契约最好尽量别动

我不同意的部分：

- 不建议把多规则的主逻辑放在 handler 外 wrapper
- 更好的切点是 dispatcher
- rule 维度最终一定要进入 message、scheduler、processor

一句话总结：

可以简单做，但不能假装它还是单配置系统。

如果要继续推进，我建议下一步直接出一版“代码级改造清单”，列清楚：

- 哪些结构要新增
- 哪些函数签名要改
- 哪些地方可以保持兼容
- 第一版先不做什么
