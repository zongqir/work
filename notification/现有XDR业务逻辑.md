# 现有 XDR 业务逻辑

## 1. 当前消息处理机制

当前 XDR 侧主要存在两种消息处理机制：

### 1.1 HTTP 处理机制

- 通过 `HTTP` 方式接入和处理消息
- 支持多渠道场景
- 当前已覆盖的渠道包括：
  - 企业微信
  - 微信
  - 邮件

这套机制的特点是接入形态相对通用，适合承接多种通知类场景。

### 1.2 微信推送机制

- 该机制从名称和实现形态上看，当前主要只支持微信推送
- 底层实现实际依赖 `MQ`
- 由于历史实现与具体业务耦合较深，当前暂时不具备大规模复用能力

这意味着它虽然不是纯同步调用模式，但目前仍更偏向“特定业务定制能力”，而不是“可通用扩展的统一推送能力”。

## 2. 当前主要业务场景

当前 XDR 的实际使用场景，主要集中在工单场景下的邮件通知：

- 工单系统在特定节点触发通知
- 通知主要通过邮件方式发送

从现状看，这里的消息触发并不是系统高频自动触发，而是较大程度依赖人工操作，因此整体 `QPS` 不高。

## 3. 触发机制

结合当前场景和 AES 侧诉求，触发机制可以先归纳为以下几类：

### 3.1 安全类触发

- 产生一个新的安全事件
- 产生一个新的安全告警

这类消息通常和安全运营直接相关，要求能够稳定触发，并支持按规则聚合后发送。

### 3.2 审计类触发

- 管理员登录
- 任务状态变更

这类消息更偏操作留痕和状态通知，通常对内容上下文要求更高，但未必要求实时逐条推送。

### 3.3 系统类触发

- 授权阈值告警，例如点数不足 `90%`
- 剩余可用时间不足 `30d`

这类消息本质上是阈值型或规则型通知，天然适合做周期检测和汇总推送。

## 4. 当前链路特点

结合现有实现，可以归纳出以下特点：

- `HTTP` 链路相对通用，适合承接多渠道通知
- 微信推送链路虽然底层依赖 `MQ`，但当前只适用于微信场景
- 微信推送链路业务耦合较强，难以直接扩展到更广泛的通知场景
- 当前主场景以人工触发为主，因此系统尚未面临明显的高并发压力

## 5. Notification 底座的现有能力与发送逻辑

从目前掌握的信息看，`notification` 本质上是 XDR 底座对外暴露的一层 `HTTP` 通知接口，其核心职责是把上游传入的通知请求，按指定渠道转发到不同下游。

### 5.1 当前定位

- 对外提供统一 `HTTP` 接口
- 按请求里指定的渠道进行路由
- 转发到不同下游通道
- 记录发送历史
- 对部分链路做基础限流

总体上，它更像一个“统一投递入口 + 渠道转发层”，而不是一个完整的通知编排平台。

### 5.2 当前真实发送逻辑

当前 `notification` 的发送逻辑可以理解为：

1. 上游业务通过 `HTTP` 调用 `notification`
2. `notification` 根据请求中指定的渠道选择对应下游
3. 按下游协议完成同步转发
4. 将发送记录写入历史表
5. 对部分场景执行限流控制
6. 同步返回调用结果给上游

因此它当前本质上仍然是一个同步调用链路，不是异步消息编排链路。

### 5.3 当前已经具备的底座能力

如果复用 `notification HTTP`，当前至少可以直接复用这些能力：

- 统一外部接入协议
- 多渠道路由转发
- 下游协议适配
- 发送历史留痕
- 部分基础限流

这些能力对 AES 是有价值的，因为它们正好覆盖了“真正发出去”这一层。

### 5.4 当前明显不具备或不适合承接的能力

结合现状，`notification` 当前不适合直接承接以下能力：

- 事件归并
- 非实时汇总发送
- 页面可配置的频率策略
- 租户级启用/禁用控制
- 复杂的模板装配和内容生成
- 大规模异步削峰和消息堆积处理

尤其是 `MQ` 能力这块，目前底座并不倾向于做重。如果 AES 后续强依赖新的 `MQ` 通知能力，大概率需要单独提需求，而不能默认底座会直接提供。

### 5.5 对 AES 的直接影响

这意味着 AES 在复用 `notification` 时，不能把它当作“通知平台全栈能力”，而应该把它当作“发送执行层”。

更合适的职责划分是：

- `AES` 负责事件识别、分类、汇总、频率控制、租户配置、内容生成
- `notification` 负责按既定渠道把最终消息同步投递到下游

也就是说，`notification` 解决的是“怎么发”，而 AES 需要解决的是“什么时候发、发什么、发给谁、走哪些渠道”。

## 6. 当前约束与问题

当前已经暴露出的核心约束主要有：

- 现有主场景偏人工触发，链路设计更适合低 `QPS` 场景
- 当前微信推送存在明显限制，单个租户一天最多只能推送 `5` 条消息
- `notification` 当前以同步 `HTTP` 转发为主，不具备完整的异步编排能力
- 底座当前并不明显倾向于新增较重的 `MQ` 通知能力
- AES 期望支持短信，但当前已明确的底座能力主要是企业微信、微信、邮件，短信能力还需要进一步确认

因此，从现状上看，当前 XDR 更像是“能够支撑特定场景的通知能力”，但距离“统一、多渠道、可规模化治理的消息推送平台”还有明显差距。

## 7. AES 场景下的特定要求

基于 AES 当前给出的触发类型、推送方式和使用原则，可以明确看出，后续方案不能只满足“能发消息”，而是需要满足以下特定要求。

### 7.1 需要支持按消息类型进行分类治理

消息至少需要支持三大类：

- 安全类
- 审计类
- 系统类

不同类型的消息，在触发规则、接收对象、推送频率、内容模板上都可能不同，因此消息模型里需要明确的分类字段，而不能只做一个通用文本通知。

### 7.2 需要支持多渠道统一编排

AES 当前明确的推送方式包括：

- 企业微信
- 邮件（`SMTP`）
- 短信

这意味着后续系统要具备统一渠道抽象能力，至少要支持：

- 同一类消息可选择一种或多种渠道发送
- 不同租户可配置不同渠道组合
- 渠道失败后具备降级、补偿或重试能力

### 7.3 需要支持“非实时、汇总发送”

AES 明确提出推送不是实时逐条发送，而是需要汇总发送。

这会直接带来几个设计要求：

- 需要有汇总窗口，例如按分钟、小时、天进行聚合
- 需要定义同类消息的合并规则
- 需要避免同一资产、同一事件在短时间内被重复轰炸
- 需要支持批量生成消息内容，而不是收到事件后立即下发

这说明系统核心不只是“推送”，还需要具备“归并和调度”能力。

### 7.4 需要支持页面可配置的推送频率

由于每类消息推送频率不一样，而且页面可配，因此后续方案必须支持：

- 按消息类型配置推送频率
- 按租户配置是否启用该类消息
- 按租户配置推送渠道
- 配置变更后能够动态生效

也就是说，频率策略不能写死在代码里，需要变成平台配置能力。

### 7.5 需要支持富文本和结构化内容

目前消息格式虽然还未完全确定，但已经能明确几个要求：

- 和资产有关的消息，需要带上资产信息
- 每类消息内容会有差异
- 企业微信不能只支持纯文字，还需要支持文字、链接、附件或图片

这意味着消息内容层至少需要具备：

- 统一消息信封
- 按消息类型扩展的业务内容
- 按渠道渲染不同格式的能力

如果没有这层抽象，后续每加一种消息类型或一种渠道，逻辑都会迅速耦合。

### 7.6 需要支持默认内置，但允许租户自主启停

AES 的原则是参考 `trend`，默认内置，但支持客户：

- 选择启用或禁用
- 选择推送方式

这进一步说明系统必须从一开始就按“租户配置化”思路设计，而不是写成固定通知逻辑。至少要支持：

- 默认策略
- 租户级覆盖策略
- 启用/禁用开关
- 渠道选择

## 8. 基于复用 Notification HTTP 的设计重点

如果当前方向是优先复用 `notification HTTP`，而不是新做一套 `MQ` 通知能力，那么 AES 这边至少需要补齐以下设计要素。

### 8.1 事件到通知任务的转换层

AES 产生的安全类、审计类、系统类事件，不能直接逐条调用 `notification`。中间需要一层转换，把原始事件沉淀成可汇总、可调度、可发送的通知任务。

这一层至少要解决：

- 事件分类
- 汇总维度
- 去重规则
- 发送时间计算
- 渠道选择

### 8.2 汇总与调度层

由于 AES 明确要求“非实时、汇总发送”，因此必须有调度层负责：

- 按频率扫描待发送事件
- 生成本次汇总消息
- 控制是否达到发送窗口
- 避免同一时间窗口重复发送

这部分能力不能依赖 `notification` 提供。

### 8.3 内容组装与渠道渲染层

AES 侧需要先生成“最终要发送的消息内容”，然后再调用 `notification`。

原因是：

- 不同消息类型内容差异大
- 不同渠道格式差异大

因此更合理的方式是 AES 输出标准化消息，再交给 `notification` 做发送执行。
这里的核心是引入模板来处理不同的消息通知。

### 8.4 发送记录与补偿层

由于 `notification` 当前整体仍是同步调用，因此 AES 不能只做一次 `HTTP` 调用就结束，还需要在自身侧保留发送任务状态，例如：

- 待发送
- 发送中
- 发送成功
- 发送失败
- 待重试

否则一旦出现超时、限流、下游失败，就很难做补偿。

### 8.5 配额与限流协同

虽然底座和下游会做部分限流，但 AES 侧仍然需要前置考虑配额问题，尤其是：

- 单租户微信每天最多 `5` 条消息

这意味着 AES 在汇总和调度时就要考虑：

- 哪些消息必须占用微信额度
- 哪些消息可以改走邮件或短信
- 哪些消息应合并后再发
- 当额度耗尽时如何降级

不能把这个问题完全留给底座发送时再兜底。

### 8.6 对 MQ 的态度

从当前信息看，底座更倾向于复用现有 `HTTP` 能力，而不是新增一套更重的 `MQ` 通知能力。

因此当前更现实的方向应该是：

- 先按 `AES 调度/汇总 + notification HTTP 同步发送` 的方式落地
- 如果后续确实出现明显的削峰、异步解耦、批量堆积处理诉求，再单独评估并提出 `MQ` 能力需求

这条路径更符合当前底座的建设倾向。

## 9. 当前现状总结

一句话概括当前 XDR 的业务现状：

- 已经具备 `HTTP` 多渠道通知能力和一套基于 `MQ` 的微信推送能力
- 但主要落地场景仍然是低 `QPS` 的工单邮件通知
- 微信链路复用性不足
- 同时还受到微信推送配额限制，即单个租户一天最多只能推送 `5` 条消息

而从 AES 场景看，后续真正需要的已经不是单纯“发出去”，而是一个支持分类、聚合、频率控制、富内容、多渠道和租户配置化的统一通知能力。

在当前前提下，更合理的落地方向不是推动底座先做重型 `MQ` 能力，而是先复用现有 `notification HTTP` 作为统一发送出口，由 AES 自身补齐通知编排和调度能力。
