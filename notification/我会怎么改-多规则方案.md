# 我会怎么改：多规则通知方案

## 目标

我要解决的问题不是“把系统做成一个复杂规则引擎”，而是：

- 同一个 `tenant + message_type`
- 支持多条规则
- 每条规则独立决定：
  - 是否实时
  - 是否聚合
  - 聚合周期
  - 过滤条件
  - 发送渠道
  - 接收人

同时我希望：

- 尽量不改业务 handler 契约
- 尽量不让复杂度蔓延到整个系统
- 把复杂度集中在“规则编译”和“dispatcher/scheduler 的 rule 维度执行”

## 核心思路

我会把系统拆成两层模型：

1. 上游配置模型
2. 运行时执行模型

上游配置模型由外部接口决定，我不强行要求它长什么样。

本系统真正消费的，是一组运行时规则：

```go
type ExecutableRule struct {
    RuleID                 string
    SourceRuleID           string
    TenantID               string
    MessageType            string
    ConfigVersion          string
    RealtimeEnabled        bool
    AggregateEnabled       bool
    AggregatePeriodMinutes int
    Filter                 json.RawMessage
    Channel                render.ChannelPolicy
}
```

这里的原则很简单：

- 上游结构可以复杂
- 进入运行时之前，统一编译成扁平规则
- 从 `dispatcher` 开始，后面的链路一律只认 `ExecutableRule`

## 为什么这么改

### 我不改 handler 契约

现有 handler 已经有足够好的边界：

- `NewFilter()`
- `Evaluate(...)`
- `Aggregate(...)`

它本来就是“给定一份 filter，判断事件是否命中 / 计算聚合结果”。

这和“多规则”并不冲突。

多规则不是业务 handler 的责任，而是平台调度责任。

所以我的原则是：

- handler 继续只处理“一条规则”
- dispatcher 负责把“一个 message_type 下的多条规则”拆开执行

### 我不把多规则逻辑塞进 handler wrapper

wrapper 不是不能做，而是不该成为主切点。

因为多规则真正影响的不只是业务判断，还影响：

- 消息身份
- 幂等 key
- 聚合调度水位
- 消费侧取渠道配置

这些都不属于 handler 职责。

## 运行时模型设计

### 1. 规则结构

我会新增一个内部运行时结构，比如放在新的文件里：

- `notification/code/config/rules.go`

结构大致如下：

```go
package config

import (
    "encoding/json"
    "work/notification/code/internal/render"
)

type ExecutableRule struct {
    RuleID                 string
    SourceRuleID           string
    TenantID               string
    MessageType            string
    ConfigVersion          string
    RealtimeEnabled        bool
    AggregateEnabled       bool
    AggregatePeriodMinutes int
    Filter                 json.RawMessage
    Channel                render.ChannelPolicy
}
```

### 2. 规则集合

我不会继续让运行时只返回一个 `MessageConfig`。

我会改成：

```go
type MessageRuleSet struct {
    TenantID      string
    MessageType   string
    ConfigVersion string
    Rules         []ExecutableRule
}
```

这个集合是运行时最重要的配置载体。

## 配置加载怎么改

### 目标

把当前：

`tenant -> message_type -> raw_config`

改成：

`tenant -> message_type -> rule_set`

### 我会怎么做

我会保留 cache 机制，但换缓存内容。

当前 `config.Cache` 缓存的是：

- `map[string]map[string]json.RawMessage`

我会改成缓存“编译后的规则集合”，例如：

```go
map[string]map[string]MessageRuleSet
```

如果为了兼容现在的 `LoadAll` 返回值，也可以先保留原始接口，然后增加一层编译器：

```go
type RuleSetLoader interface {
    LoadAllRuleSets(ctx context.Context) (map[string]map[string]MessageRuleSet, error)
}
```

实际实现里：

1. 调外部配置接口
2. 拿到原始配置
3. 转成内部 `MessageRuleSet`
4. 存入 cache

### 这一层我会强制做两件事

- 生成稳定的 `RuleID`
- 生成 `ConfigVersion`

如果上游已经提供，就直接用。

如果上游不提供稳定 rule id，我会明确要求它提供；实在不行才退化到用一组关键字段计算 hash，但这不是最佳方案。

## Dispatcher 怎么改

这里是本次改造的主战场。

### 当前问题

当前 `Dispatcher` 的设计是：

- 只加载一条配置
- 只判断一次 `realtime_enabled`
- 只判断一次 `aggregate_enabled`
- 只发一条消息

多规则下，这个模型不成立。

### 我会改成什么样

我会让 dispatcher 加载 `MessageRuleSet`，然后按 rule 遍历执行。

#### 实时链路

当前：

```go
SendRealtime(ctx, tenantID, messageType, event)
```

调用行为改为：

1. 加载这个 `tenant + message_type` 的 `rule_set`
2. 遍历所有 `RealtimeEnabled == true` 的规则
3. 对每条规则：
   - parse filter
   - 调用 `Evaluate(...)`
   - 如果命中，发布一条带 `rule_id` 的消息

我不会改 handler 接口，只会在 dispatcher 里循环。

#### 实时幂等 key

当前实时幂等 key 不带规则维度，这是不够的。

我会改成：

```text
realtime:{tenant_id}:{message_type}:{rule_id}:{biz_key}
```

这样同一事件命中两条规则时，不会互相冲掉。

#### 聚合链路

聚合我不会继续保留“按 message_type 直接调”的内部实现。

外部入口可以还是：

```go
SendAggregate(ctx, tenantID, messageType, windowStart, windowEnd)
```

但内部会变成：

1. 加载 `rule_set`
2. 遍历所有 `AggregateEnabled == true` 的规则
3. 每条规则独立执行一次 `Aggregate(...)`
4. 每条规则各自发布一条消息

如果后面 scheduler 要更细粒度控制，也可以再加：

```go
sendAggregateForRule(ctx, rule, windowStart, windowEnd)
```

#### 聚合幂等 key

我会改成：

```text
aggregate:{tenant_id}:{message_type}:{rule_id}:{window_start}:{window_end}
```

## DispatchMessage 怎么改

这个地方一定要加 rule 维度。

我会在 [contract.go](/C:/Users/Administrator/code/work/notification/code/pkg/notification/contract/contract.go:37) 的 `DispatchMessage` 里新增字段：

```go
type DispatchMessage struct {
    IdempotencyKey string       `json:"idempotency_key"`
    TenantID       string       `json:"tenant_id"`
    MessageType    string       `json:"message_type"`
    RuleID         string       `json:"rule_id"`
    ConfigVersion  string       `json:"config_version,omitempty"`
    Source         string       `json:"source"`
    RetryCount     int          `json:"retry_count"`
    CreatedAt      time.Time    `json:"created_at"`
    ExpectedSendAt time.Time    `json:"expected_send_at"`
    ExpireAt       time.Time    `json:"expire_at"`
    BizVars        TemplateVars `json:"biz_vars"`
}
```

### 为什么 `RuleID` 必须有

因为消费侧需要知道：

- 这条消息当初是按哪条规则发出来的
- 应该按哪条规则找接收人和渠道

否则你只能在消费时按 `tenant + message_type` 重新查当前配置，这在多规则和配置变更场景下都不稳。

### 为什么 `ConfigVersion` 最好有

第一版不是绝对必须，但我倾向于加。

这样即使配置改了，也可以做：

- 严格按消息发布时版本取配置
- 或至少在日志里看到“消息用的是哪个配置版本”

## Processor 怎么改

`Processor` 现在是：

1. 收到一条消息
2. 按 `tenant + message_type` 加载配置
3. 取唯一 channel
4. 渲染
5. 发送

我会改成：

1. 收到一条消息
2. 按 `tenant + message_type` 加载规则集合
3. 用 `msg.RuleID` 找到唯一规则
4. 取这条规则的 channel
5. 渲染
6. 发送

也就是说，`Processor` 不再问：

- “这个 message_type 应该发什么”

而是问：

- “这条 dispatch message 对应的 rule 是什么”

### 接口层我会这样改

当前：

```go
LoadConfig func(ctx context.Context, tenantID, messageType string) (*config.MessageConfig, error)
```

我会改成类似：

```go
LoadRuleSet func(ctx context.Context, tenantID, messageType string) (*config.MessageRuleSet, error)
```

然后在 `Process(...)` 里用 `RuleID` 查规则。

## Scheduler 怎么改

这是第二个关键点。

### 当前问题

当前聚合水位只有：

- `tenant_id`
- `message_type`

这在多规则聚合下不成立。

例如：

- 规则 A：5 分钟聚合
- 规则 B：60 分钟聚合

如果共用一个水位，必然互相覆盖。

### 我会改成什么

我会给聚合水位加 `rule_id`。

例如：

```go
type AggregateWatermark struct {
    TenantID      string
    MessageType   string
    RuleID        string
    LastWindowEnd time.Time
    UpdatedAt     time.Time
}
```

接口改成：

```go
type AggregateWatermarkStore interface {
    LastWindowEnd(ctx context.Context, tenantID, messageType, ruleID string) (time.Time, error)
    SaveWindowEnd(ctx context.Context, tenantID, messageType, ruleID string, windowEnd time.Time) error
}
```

### 调度逻辑

调度器每次 tick：

1. 加载所有 tenant 的 rule sets
2. 遍历所有 rule
3. 只处理 `AggregateEnabled == true` 且 `AggregatePeriodMinutes > 0` 的规则
4. 每条规则独立算窗口
5. 每条规则独立推进自己的 watermark

这样设计最干净。

## Handler 契约我怎么处理

我不会先改。

当前 handler 契约已经够用了：

- `NewFilter()` 提供 filter 结构
- `Evaluate(...)` 吃单条规则的 filter
- `Aggregate(...)` 吃单条规则的 filter

我只会在 dispatcher 里，对每条规则各 parse 一次 filter，然后复用现有 handler。

## Render 和 Sender 怎么处理

这里我基本不动。

因为 render / sender 本来就只依赖：

- 一条消息
- 一份渠道策略

只要 `Processor` 能通过 `RuleID` 取到正确的 channel，它们不需要感知“多规则”。

这也是我喜欢把复杂度压在 rule_set + dispatcher + scheduler 的原因。

## 第一版我会明确不做什么

为了控制复杂度，第一版我不会做这些：

- 规则优先级
- 首条命中即停止
- 规则互斥
- 规则冲突检测
- 在平台内再做一套复杂 schema
- 动态模板继承体系

第一版只支持：

- 多规则并存
- 命中即各发各的
- 聚合按规则独立跑

## 如果不考虑历史，我会怎样组织接口

如果完全不考虑历史结构，我会更倾向于直接让核心依赖 `RuleSetLoader`，而不是保留现在的 `LoadAll raw json` 风格。

例如：

```go
type RuleSetLoader interface {
    LoadRuleSet(ctx context.Context, tenantID, messageType string) (*config.MessageRuleSet, error)
    LoadAllRuleSets(ctx context.Context) (map[string]map[string]config.MessageRuleSet, error)
}
```

然后：

- `Dispatcher` 依赖 `RuleSetLoader`
- `Processor` 依赖 `RuleSetLoader`
- `Scheduler` 依赖 `LoadAllRuleSets`

这样边界最清晰。

## 我会怎么分阶段落地

### Phase 1：先打通实时多规则

目标：

- rule_set 跑通
- `DispatchMessage.RuleID` 跑通
- processor 能按 rule id 找规则

这个阶段先不改聚合 scheduler，也可以。

### Phase 2：再改聚合多规则

目标：

- 聚合按 rule 执行
- 聚合水位带 `rule_id`
- 聚合幂等 key 带 `rule_id`

### Phase 3：最后补观测和治理

目标：

- rule 命中日志
- rule 维度指标
- 配置版本可追踪

## 最终结论

如果是我来改，我不会把问题定义成“单配置系统上包一层 list handler”。

我会把它定义成：

- 配置接入层：把外部配置编译成内部规则集
- 分发层：按规则执行
- 调度层：按规则维护聚合窗口
- 投递层：按规则身份找渠道

同时坚持两条原则：

- 不改业务 handler 契约
- 不在平台里引入过重抽象

一句话总结：

我会把“多规则”做成平台内部的运行时规则模型，而不是做成 handler 的额外包装技巧。
