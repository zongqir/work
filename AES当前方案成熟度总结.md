# AES 当前方案成熟度总结

## 1. 当前判断

经过这一轮边界收敛，AES 通知方案已经从“通知编排 + 业务调度”收缩成：

- 纯通知执行与通道编排层

这次收敛非常重要，因为它直接砍掉了一大块原本不该归 AES 的责任：

- 业务巡检
- 业务汇总
- 业务定时
- 业务补跑

所以当前最合适的判断是：

- 大方向已经稳定
- 边界比之前清楚很多
- 已经可以进入偏实现的详细设计阶段

## 2. 已经明确的部分

下面这些内容，我认为已经基本确定，不需要再大幅推翻。

### 2.1 能力定位

已经明确：

- AES 不负责业务何时触发
- AES 不负责业务何时巡检
- AES 只负责通知执行与通道编排

这部分是当前方案最重要的稳定结论。

### 2.2 配置模型

已经明确：

- 默认配置放 `ConfigMap`
- 租户特殊配置放 DB override
- `GET` 返回最终生效配置
- `PUT` 只写租户 override

而且配置范围已经收敛到通知执行域：

- 启用/禁用
- 渠道
- 接收对象
- 模板绑定
- 配额/限流

### 2.3 快照模型

已经明确：

- `common + tenants` 两层快照
- 公共和租户共用同一套 body
- 不靠隐式字段表达默认/覆盖语义

### 2.4 多服务落点

已经明确：

- 业务触发仍按现有微服务边界分散存在
- 不新建一个统一业务服务去反查所有数据

也就是说，谁拥有业务动作，谁决定什么时候发起通知请求。

### 2.5 执行器归属

已经明确：

- `aes-crontab` 更适合作为统一执行器承载体
- 不建议每个业务服务各自直接发送

### 2.6 二方库边界

已经明确：

- 业务侧只需要一个 producer sdk，负责 insert batch
- 执行逻辑不再额外抽一层执行器 sdk
- 执行器内部只保留轻量 `notification client`

### 2.7 幂等方向

已经明确：

- 上游接受 at-least-once
- 下游 `notification` 负责幂等兜底
- 批次表也要有唯一键或 `upsert/insert ignore`

### 2.8 写放大优化方向

已经明确：

- 不追求过程状态尽可能完整
- 只保留关键事实落库
- 弱化中间态
- 不再把业务调度状态塞进通知系统

## 3. 已经形成共识但还需细化的部分

这些点方向已经基本对了，但实现细节还需要继续补。

### 3.1 batch 表结构

现在已经明确：

- 批次表是主事实表
- 执行器只扫批次表

但还没完全定死：

- 具体字段名
- `payload` 还是 `payload_ref`
- 索引
- 唯一键
- `upsert` 策略

### 3.2 执行器状态推进

现在已经明确：

- 不想把中间状态做太重

但还要继续收：

- 是否真的不落 `sending`
- 失败重试具体做到几次
- 失败后回到 `pending` 还是直接 `failed`

### 3.3 `notification` 契约

现在已经明确：

- AES 最终复用 `notification HTTP`

但还没完全定死：

- 请求体结构
- 企业微信/邮件/短信的公共字段
- 附件、图片、链接怎么映射

### 3.4 模板渲染策略

现在已经明确：

- 批次生成时写入稳定的 `value/payload`
- 发送时按模板渲染

但还要继续收：

- 模板变量约束
- 变量缺失怎么处理
- 模板版本如何管理

### 3.5 可观测性

现在已经明确：

- 需要按 `msgType` 看执行情况

但还要定：

- `pending_batch_count`
- `failed_batch_count`
- `oldest_pending_age`
- 渠道维度成功率

## 4. 目前暂时不建议继续深挖的部分

这些内容不是永远不考虑，而是当前阶段没必要优先做重。

### 4.1 业务调度

当前已经明确：

- 不归 AES

所以暂时不建议继续在 AES 文档里细化：

- `slot`
- `checkpoint`
- `daily_09:00`
- 业务补跑

### 4.2 MQ 化

当前已经比较明确：

- 第一版不优先做 MQ

所以暂时不建议把大量精力花在 MQ 方案细化上。

### 4.3 完整工作流状态机

当前也比较明确：

- 不做重型工作流
- 不做过多中间态

所以暂时不建议扩展成复杂状态平台。

## 5. 当前方案最重要的变化

和最开始相比，当前方案已经发生了几个结构性变化：

- 从“通知层可能承担业务调度”收缩成“纯通知执行层”
- 从“编排 -> 业务 -> 执行”收缩成“上游触发 -> AES 执行”
- 从“讨论 slot/checkpoint”收缩成“只保留通知执行必需状态”
- 从“可能反查业务数据”收缩成“上游自己组装通知请求”
- 从“时间触发是主设计点”收缩成“渠道、模板、执行是主设计点”

这些变化意味着，当前方案已经比之前更贴近真实职责边界。

## 6. 当前最适合进入详细设计的模块

如果从现在开始进入详细设计，我认为最适合优先展开的是：

1. 统一配置字段与接口
2. 批次表结构
3. `notification` 请求模型
4. 执行器扫描、发送与回写逻辑
5. 幂等 key 与唯一键规则

这些内容一旦定下来，方案基本就能进入实现阶段。

## 7. 当前结论

当前 AES 通知方案已经具备比较高的成熟度。

更准确地说：

- 边界已经稳定
- 核心 tradeoff 已经明确
- 目前不需要再回到“到底要不要负责业务调度”这种大方向摇摆

接下来真正要做的，是把通知执行链路的少数关键细节继续收口。

一句话总结：

当前方案已经从“架构摇摆期”进入“执行设计期”，最重要的方向不是再扩能力，而是守住“只做通知执行层”这个边界。
